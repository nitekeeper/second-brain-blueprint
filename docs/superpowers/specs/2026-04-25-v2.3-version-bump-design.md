# Design Spec: v2.3 Schema Version Bump

**Date:** 2026-04-25
**Status:** Approved
**Author:** nitekeeper + Claude Sonnet 4.6

---

## Problem

The Query Routing Rule redesign (v2.3) changes agent behavior on every question but ships as part of v2.2 schema. Existing users on v2.2 have no migration path to receive the fix. Existing users on v2.1 would land on the old v2.2 routing rule after migrating. A schema version bump to v2.3 gives both groups a clear upgrade path via `!! migrate`.

---

## Goal

Bump schema to v2.3, update `!! migrate` to handle v2.1→v2.3 and v2.2→v2.3 in a single command, and record the release in CHANGELOG.

---

## Files Changed

| File | Change |
|---|---|
| `template/CLAUDE.md` | Schema footer: `v2.2` → `v2.3`; startup step 7 detection: `below v2.2` → `below v2.3` |
| `template/scheduled-tasks/ops/migrate.md` | Version-aware branching: detect v2.1 vs v2.2, run appropriate steps |
| `CHANGELOG.md` | Add v2.3.0 entry |

`ROADMAP.md` — no change (v2.3 is a behavior fix, not a planned milestone).

---

## Design

### migrate.md — Version-aware branching

**Detection:** Read `Schema:` from `hot.md`. Branch on value:
- `v2.2` → run v2.2→v2.3 path
- `v2.1` or below → run v2.1→v2.3 path
- `v2.3` → announce already up to date, stop

---

#### v2.2 → v2.3 path (query routing patch only)

1. **Pre-flight:** Run `python scripts/check_deps.py --python`
   - If exit non-zero: stop, show printed instructions, do not continue.

2. **Show approval request:**
   ```
   Migration: v2.2 → v2.3

   Files UPDATED:
     CLAUDE.md  (Query Routing Rule section replaced — behavior change only, no size change)

   Files UNTOUCHED:
     wiki/  (all pages, index, log, hot preserved)
     memory.md, raw/, drafts/
     scheduled-tasks/ops/  (all ops files preserved)
     scripts/  (all scripts preserved)
     sqlite-query skill (if installed)

   Shall I proceed?
   ```

3. **On approval, execute in order:**
   a. Create `backups/` directory if absent.
   b. Copy `CLAUDE.md` → `backups/CLAUDE.md-v2.2-<YYYY-MM-DD>.bak`
   c. Read current `CLAUDE.md`. Replace the entire `## Query Routing Rule` section
      (from the heading through the end of the old Step 3 line, before `---`) with
      the v2.3 content from `blueprint/template/CLAUDE.md` (same section).
   d. Patch `hot.md`: update `Schema: v2.2` → `Schema: v2.3`
   e. Append to `wiki/log.md`:
      `## [YYYY-MM-DD] migrate | v2.2 → v2.3 — wiki-first query routing redesign`
      (≤500 chars)

4. **Confirm:**
   > "Migration complete. Query routing is now wiki-first.
   > Backup saved to `backups/CLAUDE.md-v2.2-<date>.bak` — delete when satisfied."

---

#### v2.1 → v2.3 path (full restructure + patch)

Run all existing v2.1→v2.2 steps unchanged, with two modifications:
- In step 3i: write `Schema: v2.3` instead of `Schema: v2.2` when patching `hot.md`
- In step 3j: append `## [YYYY-MM-DD] migrate | v2.1 → v2.3 — lean cold-start + wiki-first routing`

Because step 3c copies `CLAUDE.md` from `blueprint/template/CLAUDE.md`, which already contains the v2.3 Query Routing Rule, the routing patch is implicit — no extra step needed.

**Confirm message:**
> "Migration complete. Cold-start: ~7,780 → ~1,080 tokens. Query routing is now wiki-first.
> Backup saved to `backups/CLAUDE.md-v2.1-<date>.bak` — delete when satisfied."

---

### CLAUDE.md — Two changes

1. **Schema footer** (last line): `Schema version: 2.2` → `Schema version: 2.3`

2. **Startup step 7 detection**: change announcement condition from:
   > "If `hot.md`'s `Schema:` is below `v2.2`"
   to:
   > "If `hot.md`'s `Schema:` is below `v2.3`"

   Updated announcement text:
   > "Blueprint v2.3 is available — run `!! migrate` to update."

---

### CHANGELOG.md — New entry

```markdown
## v2.3.0 — 2026-04-25

### Query Routing Rule redesign

**Wiki-first waterfall:** Agent no longer leads with training knowledge.
Routing is now deterministic — no judgment call at the top of the chain:

- Step 1 — Wiki (always first, no conditions)
- Step 2 — Web Search (when wiki misses or question needs current info);
  successful results auto-ingest to `wiki/inbox/` with no user prompt
- Step 3 — Training Knowledge (fallback only); always reports
  `Confidence: N/10` with staleness caveat when score ≤ 7

**`!! migrate` updated:** Handles v2.1 → v2.3 and v2.2 → v2.3 in a single
command. v2.2 users get a lightweight patch (one section swap + Schema bump).
```

---

## Rollback

- v2.2→v2.3: rename `backups/CLAUDE.md-v2.2-<date>.bak` → `CLAUDE.md`; revert `Schema:` in `hot.md` to `v2.2`
- v2.1→v2.3: same pattern as existing v2.2 rollback — rename backup, revert `hot.md`
