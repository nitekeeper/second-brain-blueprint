### [AUD-2026-04-25-015] `README.md` describes query waterfall in the old (pre-v2.3) order

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-015 |
| **Severity** | CRITICAL |
| **Status** | RESOLVED |
| **File** | `README.md` |
| **Location** | line 74 — Key Features bullet |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `- **Query waterfall** — answers from training first, then wiki, then web search`

**Criteria**
User-facing documentation must accurately describe the active schema behaviour. `CLAUDE.md` v2.3 defines the waterfall as: Step 1 — Wiki (always first), Step 2 — Web Search, Step 3 — Training Knowledge (fallback only). `README.md` inverts this, listing training first.

**Cause**
The v2.3 Query Routing Rule redesign updated `template/CLAUDE.md` and `migrate.md` but did not cascade to `README.md`. The blueprint-sync rule (`blueprint-sync.md`, row "Schema or startup change") requires `README.md` to be updated on schema changes.

**Consequence**
A new user reading `README.md` to understand the system before setup will believe the agent answers from training knowledge first — the opposite of v2.3 behaviour. This creates incorrect expectations and may cause users to distrust wiki-first answers or misinterpret the `📋 Waterfall:` compliance line in every response.

**Recommendation**
Replace the Key Features bullet with:

```
- **Query waterfall** — wiki first, then web search, then training knowledge (fallback only, with confidence score)
```

### [AUD-2026-04-25-016] `user-guide.md` query waterfall section documents old order with stale step numbers

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-016 |
| **Severity** | CRITICAL |
| **Status** | RESOLVED |
| **File** | `user-guide.md` |
| **Location** | lines 51–55 — Asking Questions — Query section |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> ```
> 1. **High confidence** → answers directly from training knowledge
> 2. **Wiki topic** → checks `index.md`, reads relevant pages, answers with citations
> 3. **Not in wiki** → searches the web, summarizes findings, asks if you want to ingest the source
>
> After a Step 2 or Step 3 answer, the agent will ask: *"Worth filing this as an analysis page?"*
> ```

**Criteria**
The query waterfall description must match `CLAUDE.md` v2.3: Step 1 = Wiki, Step 2 = Web Search, Step 3 = Training Knowledge (fallback). `user-guide.md` shows the v2.1 order (training → wiki → web) and propagates the wrong step numbers into the filing-offer sentence.

**Cause**
Same root cause as Finding 015: the v2.3 Query Routing Rule redesign was not cascaded to `user-guide.md`. The blueprint-sync rule requires `user-guide.md` to be updated on schema changes.

**Consequence**
Users reading the primary usage guide believe Step 1 is training, Step 2 is wiki, Step 3 is web. The `📋 Waterfall:` compliance line in every agent response reports against the v2.3 definition; users cross-referencing it against this guide will misread correct agent behaviour as an error. The filing-offer guidance ("Step 2 or Step 3") will also mislead users who expect the offer after wiki answers (now Step 1).

**Recommendation**
Replace the waterfall list and filing sentence with:

```
1. **Wiki** → checks recent log, reads relevant pages via `index.md`, answers with `[[wiki link]]` citations
2. **Web search** → when wiki has nothing useful, or question needs current information; successful results auto-save to `wiki/inbox/`
3. **Training knowledge** *(fallback)* → when wiki and web both miss; answer includes `Confidence: N/10` staleness caveat

After a Step 1 (wiki synthesis) or Step 2 (web) answer, the agent will ask: *"Worth filing this as an analysis page?"*
```

### [AUD-2026-04-25-017] `user-guide.md` waterfall compliance example now labels correct behaviour as an error

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-017 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `user-guide.md` |
| **Location** | approx. line 187 — Footer Commands section |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `If you see \`Step 1\` on a wiki question, or a missing ops file read before an operation, the agent skipped a mandatory step and you can call it out immediately.`

**Criteria**
The compliance example must describe a genuinely wrong agent behaviour. In v2.3, `Step 1` is the wiki step — seeing `Step 1` in the `📋 Waterfall:` line on a wiki question is correct. The sentence now instructs users to flag correct behaviour as an error.

**Cause**
In the pre-v2.3 waterfall, Step 1 was training knowledge and Step 2 was wiki. The original example was valid: "Step 1" on a wiki question meant the agent skipped the wiki check. The v2.3 redesign renumbered the steps without updating this note.

**Consequence**
Users following this guidance will incorrectly challenge the agent when it returns a wiki answer labelled `Step 1`, eroding trust in the compliance mechanism. The waterfall compliance line is a key auditability feature; a wrong example negates its value.

**Recommendation**
Replace the example with a genuinely wrong v2.3 pattern:

```
If you see `Step 3` (training knowledge) on a question that should be in your wiki, or a missing ops file read before an operation, the agent skipped a mandatory step and you can call it out immediately.
```

### [AUD-2026-04-25-018] `conventions.md` "Filing Answers" section carries stale step labels from pre-v2.3 waterfall

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-018 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/conventions.md` |
| **Location** | section heading and body — "Filing Answers (Query Step 2 / Step 3)" |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `## Filing Answers (Query Step 2 / Step 3)`
> `After any Step 2 (wiki) or Step 3 (web) answer, ask: "Worth filing this as an analysis page?"`

**Criteria**
Agent-executed instruction files must use the step numbers defined in `CLAUDE.md`. In v2.3: Step 1 = Wiki, Step 2 = Web Search, Step 3 = Training Knowledge. `conventions.md` labels Step 2 as "wiki" and Step 3 as "web" — the pre-v2.3 mapping.

**Cause**
`conventions.md` was not updated during the v2.3 migration. `migrate.md`'s v2.1→v2.3 path copies `conventions.md` from `blueprint/template/scheduled-tasks/ops/`, but the source template was not updated when the routing rule changed.

**Consequence**
An agent reading `conventions.md` while handling a wiki-first answer (Step 1) will not find Step 1 listed as a filing trigger — only Step 2 and Step 3 are named. The agent may skip the filing-offer prompt after wiki answers. This is a behavioural regression: the most common answer path (Step 1 wiki hit) loses the analysis-page offer.

**Recommendation**
Update section heading and first sentence:

```
## Filing Answers (Query Step 1 / Step 2)

After any Step 1 (wiki synthesis) or Step 2 (web) answer, ask: "Worth filing this as an analysis page?"
```

### [AUD-2026-04-25-019] `CLAUDE.md` Step 2 decision tree missing branch for "web answers + wiki had partial info"

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-019 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/CLAUDE.md` |
| **Location** | Step 2 — Web Search bullet list |
| **Owner** | — |
| **Target Date** | — |

**Condition**
Step 2's decision tree has exactly two branches:
> `- If the result **directly answered the question** → silently save to \`wiki/inbox/\` and read \`@scheduled-tasks/ops/ingest.md\` to ingest it`
> `- If the result is **loosely related but did not answer** → skip ingest; use partial findings to inform Step 3`

**Criteria**
A waterfall decision tree must cover all reachable input combinations. The two listed branches handle: (a) web answered + wiki had nothing, and (b) web loosely answered + wiki had nothing. Neither branch covers: (c) web directly answered AND Step 1 returned partial wiki information. In case (c), combining both sources gives a richer, better-cited answer; the current rule silently ingests the web result and discards the wiki context.

**Cause**
The v2.3 redesign spec defines edge case 3 explicitly: *"Web finds answer + wiki had partial info → Ingest web result; answer using both sources with citations from each."* This case was not included in the replacement text written into `migrate.md` step c or `template/CLAUDE.md`.

**Consequence**
When Step 1 returns partial wiki information and Step 2 finds a complete web answer, the agent answers from the web result alone, dropping citations from the wiki pages it already read. Users lose cross-source traceability and the wiki's partial content is not reinforced.

**Recommendation**
Add a third bullet to Step 2's decision list:

```
- If the result **directly answered the question AND Step 1 returned partial wiki content** → ingest the web result; answer citing both the wiki pages and the web result
```
