### [AUD-2026-04-25-039] `migrate.md` v2.1→v2.3 preamble references non-existent section and uses vestigial step numbering

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-039 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.1 → v2.3 Path — opening paragraph and two modification bullets |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `Run all steps from the v2.1 → v2.2 migration (listed below), with two modifications:`
> `- **Step 3i:** write \`Schema: v2.3\` (not \`v2.2\`) when patching \`hot.md\``
> `- **Step 3j:** append \`## [YYYY-MM-DD] migrate | v2.1 → v2.3 — lean cold-start + wiki-first routing\``

There is no "v2.1 → v2.2 migration" section in `migrate.md` — the file goes directly from the v2.2→v2.3 path to the v2.1→v2.3 path. Furthermore, the steps listed directly below this preamble are already the v2.1→v2.3 migration steps with both modifications pre-applied: step i already writes `Schema: v2.3` and step j already appends the v2.1→v2.3 log message. The modification labels "Step 3i" and "Step 3j" use a numeric prefix ("3") inconsistent with the lettered step scheme (a, b, b2, c..., i, j) used in the "On approval, execute in order" section.

**Criteria**
Instructions should be unambiguous: an agent told to "run all steps from section X" must be able to locate section X. Modification references must match the step labels used in the same document.

**Cause**
The v2.1→v2.3 path was written as a layered derivation from a v2.1→v2.2 migration that may have existed in an earlier version of the file (or was planned as a separate section). When the path was consolidated into a single direct v2.1→v2.3 migration, the preamble text and modification bullets were not updated to reflect that the steps now already incorporate the modifications. The "Step 3" prefix in "Step 3i / 3j" is a vestige of a prior numbered-step scheme.

**Consequence**
An agent reading the v2.1→v2.3 section: (1) is told to "run all steps from the v2.1→v2.2 migration (listed below)" — no such section exists below; the agent would likely search the document, fail to find it, and fall back to executing the v2.1→v2.3 steps that do follow; (2) is told to apply "Step 3i" and "Step 3j" modifications — these labels don't match the lettered steps, so an agent following them literally would try to apply modifications that are already incorporated, a no-op. In practice, the pre-flight, approval request, and "On approval" sections are clear and the migration would execute correctly, but the preamble introduces unnecessary ambiguity. No data loss or functional failure risk.

**Recommendation**
Replace the confusing preamble with a direct description of the path:

Change:
```
Run all steps from the v2.1 → v2.2 migration (listed below), with two modifications:
- **Step 3i:** write `Schema: v2.3` (not `v2.2`) when patching `hot.md`
- **Step 3j:** append `## [YYYY-MM-DD] migrate | v2.1 → v2.3 — lean cold-start + wiki-first routing`

Because step 3c copies `CLAUDE.md` from `blueprint/template/CLAUDE.md`, which
already contains the v2.3 Query Routing Rule and v2.3 schema footer, the routing
patch and version bump are implicit.
```

To:
```
Direct migration path from v2.1 to v2.3 in a single step — no intermediate v2.2 stop required. Step c copies `CLAUDE.md` from `blueprint/template/CLAUDE.md`, which already contains the v2.3 Query Routing Rule and v2.3 schema footer, so no separate routing patch is needed. The steps below are the complete v2.1→v2.3 procedure.
```

### [AUD-2026-04-25-040] `audit.md` Notes token-estimate command is unusable in blueprint-authoring mode

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-040 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/audit.md` |
| **Location** | Notes section — last paragraph |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `For \`!! audit all\`, expect ~35,000–45,000 tokens of reads for the tracked files. Run \`python scripts/estimate_tokens.py blueprint/README.md blueprint/setup-guide.md blueprint/user-guide.md blueprint/troubleshooting.md blueprint/template/CLAUDE.md blueprint/template/scheduled-tasks/refresh-hot.md blueprint/template/scheduled-tasks/ops/*.md blueprint/skills/sqlite-query/*.md blueprint/skills/claude-code-enhanced/*.md blueprint/docs/audit-report-template.md\` for a live estimate. Warn the user up front if the session is already close to context limits.`

This command assumes deployed-user mode: (1) it uses `scripts/estimate_tokens.py`, which exists at `scripts/` in the deployed working folder but not in the blueprint repo root where blueprint-authoring mode runs (scripts live at `template/scripts/`); (2) it uses `blueprint/` path prefixes, which don't exist in the blueprint repo itself (files are at the repo root).

**Criteria**
The Notes section is guidance for the agent conducting an audit. `!! audit all` is primarily a blueprint-maintainer tool run in blueprint-authoring mode — the scenario where both the path prefix and the script location are wrong is the most common one.

**Cause**
The command was written for deployed-user context (where the blueprint is a `blueprint/` subdirectory and `scripts/` exists at the working folder root). Blueprint-authoring mode was added later, and this specific Note was not updated. No documented fallback for when the command fails.

**Consequence**
A blueprint maintainer who tries to run the Notes command literally in blueprint-authoring mode will get a `No such file or directory` error for `scripts/estimate_tokens.py`. The command is informational (not a required step), so the audit proceeds regardless, but the pre-flight token warning is unavailable. As a workaround, auditors have been using `chars ÷ 4` estimates manually (as seen in AUD-007 through AUD-009), but this fallback is not documented in audit.md.

**Recommendation**
Add a blueprint-authoring mode note and the correct fallback command to the Notes paragraph:

Change:
```
For `!! audit all`, expect ~35,000–45,000 tokens of reads for the tracked files. Run `python scripts/estimate_tokens.py blueprint/README.md ...` for a live estimate. Warn the user up front if the session is already close to context limits.
```

To:
```
For `!! audit all`, expect ~35,000–45,000 tokens of reads for the tracked files.
- **Deployed-user mode:** `python scripts/estimate_tokens.py blueprint/README.md blueprint/setup-guide.md blueprint/user-guide.md blueprint/troubleshooting.md blueprint/template/CLAUDE.md blueprint/template/scheduled-tasks/refresh-hot.md blueprint/template/scheduled-tasks/ops/*.md blueprint/skills/sqlite-query/*.md blueprint/skills/claude-code-enhanced/*.md blueprint/docs/audit-report-template.md`
- **Blueprint-authoring mode** (`scripts/` absent): use `chars ÷ 4` as a manual estimate, or run `python template/scripts/estimate_tokens.py README.md setup-guide.md user-guide.md troubleshooting.md template/CLAUDE.md template/scheduled-tasks/refresh-hot.md template/scheduled-tasks/ops/*.md skills/sqlite-query/*.md skills/claude-code-enhanced/*.md docs/audit-report-template.md` from the repo root.

Warn the user up front if the session is already close to context limits.
```

### [AUD-2026-04-25-041] `blueprint-sync.md` cascade rows omit `ops/reference.md` and reference a non-existent CLAUDE.md section

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-041 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/blueprint-sync.md` |
| **Location** | Cascade table — "New scheduled task" and "New skill bundle added" rows |
| **Owner** | — |
| **Target Date** | — |

**Condition**
The "New scheduled task" cascade row reads:
> `blueprint/template/scheduled-tasks/<name>.md` + `ops/audit.md` (informational parenthetical) + `ops/token-reference.md` (removed in v2.2; skip) + `setup-guide.md` + `README.md` and `user-guide.md` if user-visible + **`template/CLAUDE.md` Directory Structure** + `CHANGELOG.md`

`template/CLAUDE.md` has no "Directory Structure" section — it contains Startup, Query Routing Rule, Ops Routing, Core Rules, Response Footer, and Formats. The actual directory structure document is `ops/reference.md`, which is also absent from this cascade row.

The "New skill bundle added" row reads:
> `blueprint/skills/<skill>/` + `blueprint/user-guide.md` + `blueprint/setup-guide.md` + `blueprint/ROADMAP.md` + `ops/conventions.md` if skill introduces a new hook contract

`ops/reference.md` is also absent from this row. The reference.md blueprint/ subtree currently lists individual skill bundles (`sqlite-query/` and `claude-code-enhanced/`) — adding a new skill bundle without updating reference.md leaves the directory structure diagram stale.

**Criteria**
The blueprint-sync cascade table is the canonical list of files that must be updated for each change type. `ops/reference.md` is the document an agent reads when users ask about directory layout. Any structural addition (new scheduled task file or new skill bundle) changes the directory depicted in reference.md.

**Cause**
`ops/reference.md` was added in a later schema version (AUD-037 added it to the `!! audit all` scope; it may have been added to reference.md's own blueprint/ subtree diagram at that time as well). The blueprint-sync cascade rows for "New scheduled task" and "New skill bundle added" predate this addition and were not updated. The "template/CLAUDE.md Directory Structure" entry in the "New scheduled task" row appears to be a vestigial reference to a directory listing that CLAUDE.md once contained but no longer does.

**Consequence**
Documentation-only impact. A blueprint maintainer who adds a new scheduled task file or skill bundle and follows the cascade checklist will: (1) attempt to update a "Directory Structure" section in CLAUDE.md that does not exist (step silently skipped); (2) not be prompted to update `ops/reference.md`, leaving its directory diagram stale. A future agent consulting reference.md for structure questions would report an incomplete directory tree. The omission would surface on the next `!! audit` cycle when the auditor notices reference.md's diagram has drifted — the same failure mode as AUD-037.

**Recommendation**
In the cascade table, update both affected rows:

**"New scheduled task" row** — replace `template/CLAUDE.md` Directory Structure with `ops/reference.md`:
```
`blueprint/template/scheduled-tasks/<name>.md` + `ops/audit.md` (informational parenthetical) + `setup-guide.md` + `README.md` and `user-guide.md` if user-visible + `ops/reference.md` (blueprint/ scheduled-tasks subtree) + `CHANGELOG.md`
```
(Remove the `ops/token-reference.md (removed in v2.2; skip)` note — it adds noise with no actionable content.)

**"New skill bundle added" row** — add `ops/reference.md`:
```
`blueprint/skills/<skill>/` + `blueprint/user-guide.md` + `blueprint/setup-guide.md` + `blueprint/ROADMAP.md` + `ops/conventions.md` if skill introduces a new hook contract + `ops/reference.md` (blueprint/ skills subtree)
```
