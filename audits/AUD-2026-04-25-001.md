## Executive Summary

The blueprint's core logic, scripts, and ops files are internally consistent. The dominant issue is an **incomplete summary→snapshot terminology migration**: the recent `!! wrap`/`!! ready` redesign changed both the stored format and the display behavior, but `user-guide.md`, `setup-guide.md`, `README.md`, and `troubleshooting.md` were not fully updated. ### [AUD-2026-04-25-001] `!! ready` user-guide says display snapshot verbatim

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-001 |
| **Severity** | CRITICAL |
| **Status** | RESOLVED |
| **File** | `user-guide.md` |
| **Location** | approx. line 148 — Section: Session Memory |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> "The agent will display the summary in full (verbatim) to bring you back up to speed, then immediately wipe memory.md and confirm it's clear."

This tells users that `!! ready` will print the full raw snapshot content to the screen.

**Criteria**
`template/scheduled-tasks/ops/session-memory.md` Step 5 (Exit 1) explicitly states: "Parse the `[SNAPSHOT]` block silently to restore working context. Do NOT display the raw snapshot to the user." The snapshot is a Claude-internal machine format defined as "for Claude's use only — not for human reading." The user-facing announcement is reconstructed: "Resuming: [TASK value]. Next: [NEXT value]. Ready to continue."

**Cause**
`user-guide.md` was not updated during the summary→snapshot redesign. The old format was human-readable prose, and the old `!! ready` behavior was to display it verbatim. The redesign changed both the format and the display behavior but `user-guide.md` retained the old description.

**Consequence**
An agent following the user-guide description would expose the raw `[SNAPSHOT]` block to the user on every `!! ready` invocation. Users who read the guide expect a verbose verbatim dump but will receive a two-line "Resuming: X. Next: Y." announcement, creating confusion and loss of trust.

**Recommendation**
Replace the sentence at approx. line 148 with: "The agent will silently parse the snapshot to restore context, then announce: 'Resuming: [task]. Next: [next action]. Ready to continue.' It will then wipe `memory.md` and confirm it's clear."

### [AUD-2026-04-25-005] User-guide describes wrap output as "compressed summary" with 600-token cap

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-005 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `user-guide.md` |
| **Location** | approx. line 142 — Section: Session Memory |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> "The agent will ask if there's anything specific to include, then write a compressed summary to memory.md — capturing the current task thread, the next action, any non-obvious decisions, and files changed... The summary is capped at ~600 tokens so loading it in the next session stays lean."

**Criteria**
`session-memory.md` Step 3 defines the snapshot format with no character or token cap. `wrap.py write()` accepts stdin unrestricted. The 600-token cap was removed in the redesign.

**Cause**
`user-guide.md` retains pre-redesign prose not updated during the summary→snapshot migration.

**Consequence**
Users may expect a visible token cap and attempt to "trim" output that is simply a terse structured block. The stated "~600 token" figure is now fictitious.

**Recommendation**
Replace the paragraph at approx. line 142 with: "The agent will ask if there's anything specific to include, then write a compact context snapshot to `memory.md` capturing the task in flight, exact stopping point, next action, locked decisions, and active files. The format is machine-oriented and intentionally terse — the wiki stores permanent knowledge; the snapshot only bridges the current task thread to the next session."
