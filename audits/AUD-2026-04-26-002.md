### [AUD-2026-04-26-050] `migrate.md` v2.1→v2.3 confirm message states stale "→ ~1,080 tokens" figure, contradicting README.md and user-guide.md

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-050 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.1 → v2.3 Path — Confirm section |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `"Migration complete. Cold-start: ~7,780 → ~1,080 tokens. Query routing is now wiki-first.`

The confirm message for the v2.1→v2.3 migration path states the post-migration cold-start cost as ~1,080 tokens. Since AUD-048 was applied, both README.md and user-guide.md state the cold-start as ~2,100 tokens. The migrate.md confirm is now the only remaining file in the blueprint that shows the pre-AUD-048 figure.

**Criteria**
Blueprint internal consistency requires that all cold-start token figures across the blueprint agree. AUD-048 corrected the drift in README.md and user-guide.md but explicitly excluded the migrate.md confirm message on the grounds that it was "a historical statement about the v2.1→v2.3 improvement." That reasoning was valid when README.md and user-guide.md also said ~1,080 (pre-AUD-048). After AUD-048's fixes, those files now say ~2,100, and the migrate.md confirm is the only remaining source of ~1,080 — making it an internal contradiction rather than a historical note.

**Cause**
AUD-048's scope decision excluded migrate.md's confirm message, which was reasonable in isolation. However, applying AUD-048's fix to README.md and user-guide.md while leaving migrate.md unchanged converted a borderline "historical note" into an active inconsistency: a user who migrates from v2.1, reads the confirm message (~1,080), and then reads user-guide.md (~2,100) encounters two different official figures for the same metric.

**Consequence**
A user migrating from v2.1 today receives the confirmation "Cold-start: ~7,780 → ~1,080 tokens" — approximately half the actual ~2,100 post-migration cost. If they budget sessions against ~1,080, they will find their actual costs roughly 2× higher than stated. The inconsistency also undermines trust in the blueprint's token-awareness guidance, which user-guide.md introduces as a key feature. No functional breakage; the migration executes correctly regardless.

**Recommendation**

In `template/scheduled-tasks/ops/migrate.md`, update the v2.1→v2.3 Confirm message:

Change:
```
"Migration complete. Cold-start: ~7,780 → ~1,080 tokens. Query routing is now wiki-first.
```

To:
```
"Migration complete. Cold-start: ~7,780 → ~2,100 tokens. Query routing is now wiki-first.
```

### [AUD-2026-04-26-051] Both migration paths overwrite `refresh-hot.md` without a backup step; rollback instructions omit `refresh-hot.md` restoration

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-051 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.2→v2.3 Path (step c4; Rollback section); v2.1→v2.3 Path (step g2; Rollback section) |
| **Owner** | — |
| **Target Date** | — |

**Condition**

v2.2→v2.3 path backup steps (b, b2, b3) and corresponding overwrite steps (c, c2, c3):
> ```
> b.  Copy `CLAUDE.md` → `backups/CLAUDE.md-v2.2-<YYYY-MM-DD>.bak`
> b2. Copy `scheduled-tasks/ops/conventions.md` → `backups/conventions.md-v2.2-<YYYY-MM-DD>.bak`
> b3. Copy `scheduled-tasks/ops/blueprint-sync.md` → `backups/blueprint-sync.md-v2.2-<YYYY-MM-DD>.bak`
> c4. Copy `blueprint/template/scheduled-tasks/refresh-hot.md` → `scheduled-tasks/refresh-hot.md`
> ```

Step c4 overwrites `scheduled-tasks/refresh-hot.md` with no preceding backup step (b4). The v2.1→v2.3 path has the same gap — step g2 overwrites `scheduled-tasks/refresh-hot.md` without a backup.

v2.2→v2.3 Rollback section:
> `Rename backups/CLAUDE.md-v2.2-<date>.bak → CLAUDE.md; rename backups/conventions.md-v2.2-<date>.bak → scheduled-tasks/ops/conventions.md; rename backups/blueprint-sync.md-v2.2-<date>.bak → scheduled-tasks/ops/blueprint-sync.md; revert Schema: in hot.md to v2.2.`

`scheduled-tasks/refresh-hot.md` is not listed. v2.1→v2.3 rollback has the same omission.

**Criteria**
The migration op establishes a consistent backup-before-overwrite pattern: every file overwritten by the migration (CLAUDE.md, conventions.md, blueprint-sync.md) has a matching backup step (b, b2, b3) executed before the overwrite. Rollback instructions reference only files that were backed up. `refresh-hot.md` is a user-deployed file that is overwritten by both migration paths (c4, g2) but follows neither rule: no backup is created and rollback does not include restoration.

**Cause**
AUD-047 added the copy steps for `refresh-hot.md` (c4 and g2) to fix the propagation gap. The fix was focused on propagation correctness and did not include the corresponding backup step and rollback update. The pattern of (backup step → overwrite step → rollback entry) for each overwritten file was not explicitly checked when the AUD-047 fix was applied.

**Consequence**
A user who runs `!! migrate` and then decides to roll back cannot restore their pre-migration `scheduled-tasks/refresh-hot.md` through the documented rollback procedure — no backup exists. The practical impact is low: the v2.3 refresh-hot.md is a superset of v2.2 (it adds one detection check without removing any), so it behaves correctly in a rolled-back v2.2 context. However, the rollback guarantee is incomplete and the migration op's safety pattern is inconsistent.

**Recommendation**

**v2.2→v2.3 path:** Add a backup step before c4 and update the Rollback section.

1. Add step b4 immediately before c4:
   ```
   b4. Copy `scheduled-tasks/refresh-hot.md` → `backups/refresh-hot.md-v2.2-<YYYY-MM-DD>.bak`
   ```

2. Update the Rollback section to include:
   ```
   rename backups/refresh-hot.md-v2.2-<date>.bak → scheduled-tasks/refresh-hot.md;
   ```
   (insert after the blueprint-sync.md rename)

3. Update the approval request "Files UPDATED" entry for `scheduled-tasks/refresh-hot.md` to add "(backed up to backups/)" in parentheses for consistency with how other overwritten files are described.

**v2.1→v2.3 path:** Add a backup step before g2 and update the Rollback section.

1. Add step g1.5 (or rename g2 and insert before it):
   ```
   g1.5. Copy `scheduled-tasks/refresh-hot.md` → `backups/refresh-hot.md-v2.1-<YYYY-MM-DD>.bak`
   ```

2. Update the v2.1→v2.3 Rollback section to include:
   ```
   rename backups/refresh-hot.md-v2.1-<date>.bak → scheduled-tasks/refresh-hot.md;
   ```
