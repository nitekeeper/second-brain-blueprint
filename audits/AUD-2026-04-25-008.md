### [AUD-2026-04-25-032] `README.md` approval exceptions list omits `!! audit`

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-032 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `README.md` |
| **Location** | Key Features bullet — "Approval before every wiki write" line |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `Approval before every wiki write — !! wrap and !! ready are the only exceptions (each gated by built-in safeguards)`

This bullet asserts exactly two approval exceptions, but the schema documents three.

**Criteria**
`CLAUDE.md` Core Rules lists three explicit exceptions: "`!! wrap`, `!! ready`, `!! audit`". `user-guide.md` Approval Rule section repeats the same three ("Documented exceptions (no separate approval prompt): `!! wrap`, `!! ready`, and `!! audit`"). `audit.md` step 4e confirms: "Writing the audit report is an implicit side-effect of `!! audit` — no separate approval is needed (even for a clean audit)." `README.md` omits the third.

**Cause**
When `!! audit` was granted implicit-approval status (adding the step 4e note to `audit.md` and updating `user-guide.md`), the corresponding `README.md` Key Features bullet was not updated.

**Consequence**
Documentation-only impact. A reader relying on `README.md` for the approval policy will expect the agent to pause and request explicit approval before writing the audit report — the agent will not, and the user may be confused or file a false bug report. No behavioral system failure; wiki state is unaffected.

**Recommendation**
Update the README.md bullet to name all three exceptions:

Change:
```
Approval before every wiki write — `!! wrap` and `!! ready` are the only exceptions (each gated by built-in safeguards)
```

To:
```
Approval before every wiki write — `!! wrap`, `!! ready`, and `!! audit` are the only exceptions (`!! audit` writes its report as an implicit side-effect; `!! wrap` and `!! ready` have additional built-in safeguards)
```

### [AUD-2026-04-25-033] `docs/audit-report-template.md` schema version hardcoded as v2.2

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-033 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `docs/audit-report-template.md` |
| **Location** | Report Header table — Schema Version row and Auditor row |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `| **Schema Version** | v2.2 |`
> `| **Auditor** | LLM Wiki Agent — audit.md v2.2 |`

Both rows reference `v2.2` as the schema version. The current schema is `v2.3`.

**Criteria**
`audit.md` step 4c instructs the agent to fill the Schema Version field by reading the live value from `CLAUDE.md`'s footer — so the template's stale value does not corrupt generated reports. However, the template is the canonical reference for the audit report format. A developer consulting it to understand the expected output will see `v2.2` and may incorrectly conclude that is the current schema. The same class of issue was addressed in AUD-022 (directory tree omission) and AUD-031 — keeping reference artifacts current prevents maintainer confusion.

**Cause**
The schema bumped from v2.2 to v2.3 (as part of the query-routing-rule redesign). The generated audit reports are correctly auto-populated with v2.3. The template's example scaffolding was not updated to match.

**Consequence**
No runtime impact — the agent substitutes the live schema version from `CLAUDE.md` at report-generation time, so every generated audit report correctly reads v2.3. The consequence is maintainer confusion only: a developer who uses the template as a reference, checks the example values, or diffs the template against a generated report will encounter a version mismatch that has no operational explanation.

**Recommendation**
Update both rows in `docs/audit-report-template.md`:

| Field | Current | Fix |
|---|---|---|
| `**Schema Version**` | `v2.2` | `v2.3` |
| `**Auditor**` | `LLM Wiki Agent — audit.md v2.2` | `LLM Wiki Agent — audit.md v2.3` |
