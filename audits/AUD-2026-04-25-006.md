### [AUD-2026-04-25-027] `user-guide.md` audit cost estimate not updated after AUD-026 added CHANGELOG.md to scope

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-027 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `user-guide.md` |
| **Location** | Token Awareness table ‚Äî `!! audit all` row |
| **Owner** | ‚Äî |
| **Target Date** | ‚Äî |

**Condition**
> `| Audit all (full blueprint) | ~30,000‚Äì47,000 |`

The Token Awareness table in `user-guide.md` still shows the pre-CHANGELOG.md range of `~30,000‚Äì47,000` for `!! audit all`.

**Criteria**
When `audit.md`'s token estimate or scope changes, the Blueprint Sync Rule row "Operation step change" requires `blueprint/user-guide.md` to be updated. AUD-2026-04-25-026 updated `audit.md`'s Notes line to `~60,000‚Äì70,000` and added CHANGELOG.md to the estimate command. The same figure in `user-guide.md`'s Token Awareness table was not updated. The two sources are now contradictory: a user reading `user-guide.md` before deciding whether to run `!! audit all` is shown a 47,000-token ceiling while the actual cost exceeds 60,000 tokens.

**Cause**
AUD-026's fix targeted `audit.md` only. The AUD-026 recommendation did not explicitly list `user-guide.md` as a cascade target, and the Blueprint Sync Rule was not consulted for the downstream propagation. Same cascade-miss class as AUD-026 itself.

**Consequence**
A user estimating whether their session has enough headroom to run `!! audit all` will use the wrong figure (`~47,000` vs the actual `~60,000‚Äì70,000`). An agent following `audit.md`'s updated advisory will warn users correctly, but a user who pre-checks against the cost table in `user-guide.md` before invoking the audit will underestimate by roughly 15,000‚Äì23,000 tokens. In a session with 40,000‚Äì50,000 tokens already consumed, this difference determines whether the audit can complete or will exhaust the context window mid-pass.

**Recommendation**
Update the `!! audit all` row in `user-guide.md`'s Token Awareness table:

```
| Audit all (full blueprint) | ~60,000‚Äì70,000 (CHANGELOG.md alone accounts for ~30,000+ tokens and grows with every audit cycle) |
```

### [AUD-2026-04-25-028] `migrate.md` approval requests claim `hot.md` "preserved" but both migration paths patch it

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-028 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.2‚Üív2.3 approval request "Files UNTOUCHED" block; v2.1‚Üív2.3 approval request "Files UNTOUCHED" block |
| **Owner** | ‚Äî |
| **Target Date** | ‚Äî |

**Condition**

Both migration approval requests contain:
> ```
> Files UNTOUCHED:
>   wiki/  (all pages, index, log, hot preserved)
> ```

v2.2‚Üív2.3 step e:
> `Patch \`hot.md\`: update \`Schema: v2.2\` ‚Üí \`Schema: v2.3\``

v2.1‚Üív2.3 step i:
> `Patch \`hot.md\`: Add \`Python: [python | python3]\` line (detect by running...) + Update \`Schema: v2.1\` ‚Üí \`Schema: v2.3\``

Both approval requests claim that `wiki/hot.md` is "preserved" (untouched) while the immediately following execution steps modify it.

**Criteria**
The Core Rules in `CLAUDE.md` require that every approval request disclose a complete list of all files to be created, edited, or deleted before writing anything. `hot.md` lives at `wiki/hot.md` (confirmed in `reference.md`'s directory tree). Listing it in a "Files UNTOUCHED" block while a subsequent step explicitly patches it is a direct violation of this disclosure requirement.

**Cause**
`hot.md` patching (step e / step i) is a standard, small-footprint side-effect of a Schema version bump. The approval request templates were written focusing on CLAUDE.md and ops-file changes and did not include `hot.md` in the UPDATED block. The v2.2‚Üív2.3 path's approval request was partially corrected in AUD-024 (to add `conventions.md`) but `hot.md` was not addressed at that time.

**Consequence**
A user reviewing the approval request before confirming `!! migrate` is told `hot.md` is untouched. The migration then patches `hot.md` without disclosed approval. For v2.1‚Üív2.3 the impact is larger: step i not only bumps the Schema field but also adds the new `Python:` line ‚Äî this is a non-trivial structural addition to a file the user was told would not be touched. The Core Rules' transparency guarantee is broken.

**Recommendation**

For v2.2‚Üív2.3: Move `hot.md` from the UNTOUCHED block to UPDATED:
```
Files UPDATED:
  CLAUDE.md  (Query Routing Rule section replaced ‚Äî behavior change, no size change)
  scheduled-tasks/ops/conventions.md  (Filing Answers step labels updated to v2.3 numbering)
  wiki/hot.md  (Schema: v2.2 ‚Üí v2.3)
```
And change the UNTOUCHED block to:
```
Files UNTOUCHED:
  wiki/  (all pages, index, log preserved)
```

For v2.1‚Üív2.3: Add `wiki/hot.md` to the UPDATED block:
```
Files UPDATED:
  ...
  wiki/hot.md  (Schema bumped to v2.3; Python: field added)
```
And change the UNTOUCHED block to:
```
Files UNTOUCHED:
  wiki/  (all pages, index, log preserved)
```

### [AUD-2026-04-25-029] `migrate.md` rollback procedures don't revert `conventions.md`, reintroducing AUD-018's behavioral regression

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-029 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | Rollback section ‚Äî v2.2‚Üív2.3 and v2.1‚Üív2.3 rollback instructions |
| **Owner** | ‚Äî |
| **Target Date** | ‚Äî |

**Condition**

v2.2‚Üív2.3 rollback:
> `- **v2.2‚Üív2.3:** Rename \`backups/CLAUDE.md-v2.2-<date>.bak\` ‚Üí \`CLAUDE.md\`; revert \`Schema:\` in \`hot.md\` to \`v2.2\`.`

v2.1‚Üív2.3 rollback:
> `- **v2.1‚Üív2.3:** Rename \`backups/CLAUDE.md-v2.1-<date>.bak\` ‚Üí \`CLAUDE.md\`; revert \`Schema:\` in \`hot.md\` to \`v2.1\`. All new ops files and scripts/ are inert without the v2.1 \`CLAUDE.md\` referencing them.`

Neither rollback procedure mentions reverting `conventions.md`.

**Criteria**
A rollback must restore the installation to a state functionally equivalent to the pre-migration version. Both migration paths modify `conventions.md`:
- v2.2‚Üív2.3 step c2 copies the v2.3 `conventions.md` (with "Step 1 / Step 2" labels) over the user's v2.2 file
- v2.1‚Üív2.3 step g copies the v2.3 `conventions.md` as part of its ops-file batch copy

After rollback, `CLAUDE.md` is restored to the pre-migration version (v2.2 routing: Step 1=training, Step 2=wiki, Step 3=web; or v2.1's equivalent). But `conventions.md` retains its v2.3 text: "## Filing Answers (Query Step 1 / Step 2)" and "After any Step 1 (wiki synthesis) or Step 2 (web) answer, ask: 'Worth filing this as an analysis page?'" Under the restored v2.2/v2.1 CLAUDE.md routing, the step numbers mean something entirely different (Step 1=training, Step 2=wiki/Step 3=web), producing exactly the mismatch that AUD-2026-04-25-018 diagnosed and fixed ‚Äî now silently reintroduced via incomplete rollback.

**Cause**
The rollback section was written to address only the primary migration artifact (CLAUDE.md backup) and the version field (hot.md Schema). AUD-024's fix added the `conventions.md` copy step (c2/g) after the rollback section was written, and the rollback section was not updated to match.

**Consequence**
A user who rolls back from v2.3 to v2.2 (or v2.1) will have a CLAUDE.md with the old routing logic but a `conventions.md` whose Filing Answers section uses v2.3 step numbers. An agent reading conventions.md in the rolled-back environment will interpret "Step 1 (wiki synthesis) or Step 2 (web)" using the pre-v2.3 step mapping, misidentifying the filing trigger: it will offer to file after training-knowledge answers (Step 1 in v2.2 routing) and wiki answers (Step 2 in v2.2 routing), while missing web answers (Step 3 in v2.2 routing). The backup for `conventions.md` was not taken during migration (only `CLAUDE.md` was backed up), so the user cannot restore the original file automatically.

**Recommendation**
Both rollback procedures should instruct the user that `conventions.md` must also be reverted. Since `conventions.md` was not backed up during migration, add a note with the manual revert path:

For v2.2‚Üív2.3 rollback, append:
```
  Then restore conventions.md from the blueprint template's previous version or manually revert
  scheduled-tasks/ops/conventions.md ‚Äî the Filing Answers section heading should read
  "Filing Answers (Query Step 2 / Step 3)" and the body "After any Step 2 (wiki) or Step 3 (web) answer..."
  (If unsure of the v2.2 text, run `!! migrate` again to get a clean v2.3 state rather than running with a mismatched conventions.md.)
```

For v2.1‚Üív2.3 rollback, add a similar note:
```
  Note: conventions.md was updated during migration (step g) and is NOT reverted by restoring CLAUDE.md alone.
  The v2.3 conventions.md is harmless if left in place under v2.1 CLAUDE.md since v2.1 CLAUDE.md does not embed
  explicit step-number labels for the filing rule ‚Äî but verify the filing behavior in a test session.
```

Longer-term fix: consider adding `conventions.md` to the backup step (alongside `CLAUDE.md`) so rollback can be fully automated.
