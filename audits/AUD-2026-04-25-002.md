### [AUD-2026-04-25-012] Token table has duplicate rows for `!! ready` cold-start scenario

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-012 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `user-guide.md` |
| **Location** | approx. lines 222 and 230 — Section: Token Awareness |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> Line 222: `Cold start with !! ready (full memory) | ~1,830`
> Line 230: `!! ready with full memory | ~1,830`

Two separate table rows describe the same scenario — starting a session with a loaded snapshot — and now both show the same value (`~1,830`) after the AUD-002 fix reconciled them.

**Criteria**
A token-awareness table must not contain rows that appear to measure different scenarios but produce identical values. The duplication implies a distinction that does not exist, confusing users trying to plan session budgets.

**Cause**
The "!! ready with full memory" row was added during a schema re-calibration and carried a different value (1,680) than the existing "Cold start with !! ready (full memory)" row (1,830). AUD-002's fix correctly reconciled both to 1,830, which exposed the underlying duplication — now both rows describe the same event with identical values.

**Consequence**
Users reading the token table see two rows for what appears to be the same operation and cannot determine which applies. A cautious reader might double-count tokens. A confused reader might assume the rows measure different things and search for a distinction that does not exist.

**Recommendation**
Remove the `!! ready with full memory | ~1,830` row (approx. line 230). The `Cold start with !! ready (full memory) | ~1,830` row (approx. line 222) is the more complete and accurate label. If the two rows were intended to measure different things (e.g., total startup cost vs. just the memory load cost), the distinction must be documented and the values differentiated.

### [AUD-2026-04-25-013] `migrate.md` lists `ingest.md` in both UPDATED and UNTOUCHED

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-013 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | approx. lines 52 and 59 — Files UPDATED/UNTOUCHED block |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> Line 52 (UPDATED): `scheduled-tasks/ops/ingest.md      (bash → Python, post-op advisory added)`
> Line 59 (UNTOUCHED): `ingest.md (beyond advisory update)`

`ingest.md` appears in both mutually exclusive lists.

**Criteria**
UPDATED and UNTOUCHED are mutually exclusive states. A migration manifest cannot claim the same file is both modified and left alone. This is the same rule that AUD-2026-04-25-010 enforced for `conventions.md`.

**Cause**
When AUD-010 was applied, `conventions.md` was correctly removed from the UNTOUCHED list, but `ingest.md` — which has the same dual-listing problem — was left in place. The original UNTOUCHED line read `"scheduled-tasks/ops/conventions.md, ingest.md* (beyond advisory update)"`. Only conventions.md was removed; ingest.md and its explanatory parenthetical were preserved.

**Consequence**
An agent or user reading the migration approval request will see contradictory information about `ingest.md`. An agent could skip updating it (based on UNTOUCHED) or attempt to handle it twice (based on both rows). The parenthetical "(beyond advisory update)" does not clarify the contradiction — it suggests part of the file is UNTOUCHED, which conflicts with the UPDATED entry describing the full bash→Python rewrite.

**Recommendation**
Remove `ingest.md (beyond advisory update)` from the UNTOUCHED block entirely. The corrected UNTOUCHED block should read:

```
Files UNTOUCHED:
  wiki/  (all pages, index, log, hot preserved)
  memory.md, raw/, drafts/
  sqlite-query skill (if installed)
```

### [AUD-2026-04-25-014] `audit.md` estimate command missing `claude-code-enhanced` skill files

|---|---|
| **ID** | AUD-2026-04-25-014 |

| **Location** | approx. line 93 — Notes section |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `Run python scripts/estimate_tokens.py blueprint/README.md ... blueprint/skills/sqlite-query/*.md for a live estimate.`

The `!! audit all` scope (defined in the same file, lines 24–26) includes "Every file under `blueprint/skills/` (skill bundles — recurse into subdirectories)." The estimate command shows only `blueprint/skills/sqlite-query/*.md`, omitting `blueprint/skills/claude-code-enhanced/*.md` (2 files: SKILL.md and slash-commands.md).

**Criteria**
Informational accuracy — the estimate command should match the declared audit scope. A user or agent following the notes would run a command that underestimates the true token cost of the audit.

**Cause**
The notes were written when the `sqlite-query` skill was the only installed skill. The `claude-code-enhanced` skill was added later without updating the example estimate command.

**Consequence**
The estimated token count is ~500–700 tokens lower than the actual scope requires. The stated range of "~30,000–47,000 tokens" absorbs this margin, so there is no functional breakage — it is inaccurate guidance only.

**Recommendation**
Update the example command to:
