### [AUD-2026-04-26-052] `ROADMAP.md` referenced in `reference.md` tree and `blueprint-sync.md` cascade but file does not exist

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-052 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/reference.md`, `template/scheduled-tasks/ops/blueprint-sync.md` |
| **Location** | reference.md — blueprint/ directory tree; blueprint-sync.md — "New skill bundle added" cascade row |
| **Owner** | — |
| **Target Date** | — |

**Condition**

`reference.md` blueprint/ directory tree:
```
│   ├── ROADMAP.md
```

`blueprint-sync.md` "New skill bundle added" cascade row:
> `blueprint/skills/<skill>/ + blueprint/user-guide.md + blueprint/setup-guide.md + blueprint/ROADMAP.md + ops/conventions.md if skill introduces a new hook contract + ops/reference.md (blueprint/ skills subtree)`

`ROADMAP.md` does not exist in the repository. No commit in the project history creates this file — AUD-031's fix commit ("fix: AUD-031 — add ROADMAP.md to reference.md blueprint/ directory tree") only updated the diagram in reference.md.

**Criteria**

The directory tree in `reference.md` is intended to reflect the actual file layout of the deployed blueprint. The cascade table in `blueprint-sync.md` is intended to enumerate all files that must be updated when a specific type of change is made. Both are authoritative references: the tree for navigation, the cascade for correctness assurance. A file listed in either source that does not exist is a factual error that degrades trust in both references.

**Cause**

AUD-031 (from AUD-2026-04-25-007) found that "reference.md blueprint/ tree omits ROADMAP.md." The fix resolved the finding by adding ROADMAP.md to the reference.md diagram. However, the fix was incomplete: it updated the documentation to reflect a file that was only planned (not yet created), and the "New skill bundle added" cascade in blueprint-sync.md was updated in parallel to include ROADMAP.md as a cascade target. No subsequent audit cycle caught the missing file because the fixes were marked RESOLVED and the phantom ROADMAP.md was not independently verified.

**Consequence**

Two operational failures result:

1. **Navigation mismatch:** A maintainer reading `reference.md` to orient themselves expects `ROADMAP.md` to exist in the blueprint root. Looking for it will produce confusion or a silent miss.

2. **Broken cascade for new skill additions:** Any developer who follows the "New skill bundle added" cascade in `blueprint-sync.md` is directed to update `blueprint/ROADMAP.md`. Since the file doesn't exist, the cascade step either silently fails (if the developer skips it, the blueprint's roadmap remains undocumented), creates a new ROADMAP.md from scratch without guidance, or surfaces as a confusing error. Future skill additions cannot fully honor the cascade as written.

**Recommendation**

Create `ROADMAP.md` at the blueprint root with content that matches its documented purpose (a high-level roadmap of the project). A minimal placeholder that acknowledges the existing skill bundles and documents planned future capabilities is sufficient. Once created, both `reference.md` and `blueprint-sync.md` become accurate without further changes to those files.

Example minimum content:
```markdown
# Blueprint Roadmap

## Current Skills
- `sqlite-query` — SQLite query layer (replaces grep for wikis 500+ pages)
- `claude-code-enhanced` — Native slash commands for Claude Code CLI users

## Planned / Considering
*(Add future skill or feature plans here)*
```

### [AUD-2026-04-26-053] `session-hygiene.md` `!![op]` placeholder produces `!!ingest` / `!!lint` / `!!audit` — missing standard command space

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-053 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/session-hygiene.md` |
| **Location** | "Intercept message" block; "Post-op advisory block" section |
| **Owner** | — |
| **Target Date** | — |

**Condition**

Intercept message template:
> `⚠️  A !![op] operation completed earlier in this session.`

Post-op advisory block template:
> `⚠️  Session advisory: This session has completed a !![op] operation and the`

Substitution instruction: "Replace `[op]` with the op that set SESSION_HEAVY (ingest / lint / audit)."

After substitution with `op = "ingest"`: `!!ingest`. Standard format throughout the blueprint is `!! ingest` (with a space).

**Criteria**

Every ops file, user-guide.md, CLAUDE.md, and setup-guide.md consistently uses `!! command` with a space between `!!` and the command name (e.g. `!! ingest`, `!! lint`, `!! audit`, `!! wrap`, `!! ready`). The session-hygiene.md templates deviate from this convention.

**Cause**

The `[op]` placeholder was designed to be substituted with just the op word ("ingest"), and `!!` was hard-coded immediately adjacent to it. Adding a space between `!!` and `[op]` in the template text would produce the correct `!! ingest` after substitution, but the space was omitted when the template was authored.

**Consequence**

Users who see the session advisory message will read `!!ingest` instead of `!! ingest`. The advisory remains understandable and functional — users can respond with `!! proceed` regardless — but the inconsistency is visible in a message specifically designed to represent polished, trustworthy guidance from the agent.

**Recommendation**

In `template/scheduled-tasks/ops/session-hygiene.md`, add a space between `!!` and `[op]` in both templates:

In the Intercept message, change:
```
⚠️  A !![op] operation completed earlier in this session.
```
To:
```
⚠️  A !! [op] operation completed earlier in this session.
```

In the Post-op advisory block, change:
```
⚠️  Session advisory: This session has completed a !![op] operation and the
```
To:
```
⚠️  Session advisory: This session has completed a !! [op] operation and the
```

### [AUD-2026-04-26-054] `audit.md` step 0 file-check references `scripts/` without blueprint-authoring mode fallback

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-054 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/audit.md` |
| **Location** | Step 0 — "Load prior audit context" |
| **Owner** | — |
| **Target Date** | — |

**Condition**

Step 0 instruction:
> `Check whether an \`audits/\` directory exists (\`python scripts/file_check.py audits/\`).`

Note section (bottom of file):
> `**Blueprint-authoring mode** (\`scripts/\` absent): use \`chars ÷ 4\` as a manual estimate, or run \`python template/scripts/estimate_tokens.py ...\` from the repo root.`

The Note section explicitly states `scripts/` is absent in blueprint-authoring mode and provides a fallback — but only for the token estimation command. Step 0's `python scripts/file_check.py audits/` call has no corresponding blueprint-authoring mode fallback.

**Criteria**

The Note section establishes a pattern: when `scripts/` is absent in blueprint-authoring mode, an alternative command is provided. This pattern is applied to the token estimation step but not to the file-check in step 0, creating an inconsistency within the same document.

**Cause**

The blueprint-authoring mode note was added specifically in the context of the Appendix token-estimation step (step 4c), where the need for an alternative command was immediately obvious. Step 0's use of `file_check.py` was not reviewed for the same inconsistency when the note was added.

**Consequence**

A strict agent executing `!! audit all` from the blueprint repository root (blueprint-authoring mode) encounters `python scripts/file_check.py audits/` at step 0 and gets a path error (`scripts/` doesn't exist; the script lives at `template/scripts/`). The agent can work around this by using `ls audits/` or `python template/scripts/file_check.py audits/`, but neither workaround is documented. The audit may proceed correctly via agent adaptation, but the gap makes the rule incomplete for the mode that blueprint maintainers actually use.

**Recommendation**

In `template/scheduled-tasks/ops/audit.md`, extend the step 0 parenthetical to include a blueprint-authoring mode fallback:

Change:
```
Check whether an `audits/` directory exists (`python scripts/file_check.py audits/`).
```
To:
```
Check whether an `audits/` directory exists (`python scripts/file_check.py audits/`; blueprint-authoring mode: `python template/scripts/file_check.py audits/` or `ls audits/`).
```
