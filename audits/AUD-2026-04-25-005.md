### [AUD-2026-04-25-023] `migrate.md` v2.2→v2.3 step c missing AUD-019 Step 2 third branch

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-023 |
| **Severity** | CRITICAL |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.2→v2.3 path, step c, Step 2 block |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> ```
> **Step 2 — Web Search**
> Runs when: (a) wiki returned nothing useful, OR (b) question needs current or recent information.
> - Search and summarize
> - If the result **directly answered the question** → silently save to `wiki/inbox/` and read `@scheduled-tasks/ops/ingest.md` to ingest it
> - If the result is **loosely related but did not answer** → skip ingest; use partial findings to inform Step 3
> ```

The replacement block in `migrate.md` step c has only two Step 2 branches; `template/CLAUDE.md` (after the AUD-019 fix) has three.

**Criteria**
The v2.2→v2.3 migration must produce a `CLAUDE.md` functionally identical to a fresh-setup install from `template/CLAUDE.md`. AUD-2026-04-25-019 added a third Step 2 branch to `template/CLAUDE.md`: `"If the result **directly answered the question AND Step 1 returned partial wiki content** → ingest the web result; answer citing both the wiki pages and the web result"`. This branch is absent from migrate.md's replacement block.

**Cause**
AUD-019 fixed `template/CLAUDE.md` but the fix was not cascaded to `migrate.md`'s verbatim replacement block. The Blueprint Sync Rule does not explicitly list `migrate.md` as a cascade target for Query Routing Rule changes (see also Finding AUD-2026-04-25-025).

**Consequence**
Any user currently on v2.2 who runs `!! migrate` will receive a `CLAUDE.md` with an incomplete Step 2 decision tree. When their wiki has partial information on a topic and a web search directly answers the question, the agent will take the first branch (ingest only, no wiki citation) instead of the third branch (ingest + cite both sources). Fresh-setup users and migrated users will have divergent routing behavior — two classes of "v2.3" installations with different rules.

**Recommendation**
Insert the missing branch in the step c replacement block between the "directly answered" and "loosely related" bullets:

```
- If the result **directly answered the question AND Step 1 returned partial wiki content** → ingest the web result; answer citing both the wiki pages and the web result
```

Full corrected Step 2 block for step c:

```
**Step 2 — Web Search**
Runs when: (a) wiki returned nothing useful, OR (b) question needs current or recent information.
- Search and summarize
- If the result **directly answered the question** → silently save to `wiki/inbox/` and read `@scheduled-tasks/ops/ingest.md` to ingest it
- If the result **directly answered the question AND Step 1 returned partial wiki content** → ingest the web result; answer citing both the wiki pages and the web result
- If the result is **loosely related but did not answer** → skip ingest; use partial findings to inform Step 3
```

### [AUD-2026-04-25-024] `migrate.md` v2.2→v2.3 doesn't re-copy `conventions.md` (AUD-018 cascade miss)

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-024 |
| **Severity** | CRITICAL |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.2→v2.3 path — approval request "Files UNTOUCHED" block; step c list |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> ```
> Files UNTOUCHED:
>   wiki/  (all pages, index, log, hot preserved)
>   memory.md, raw/, drafts/
>   scheduled-tasks/ops/  (all ops files preserved)
>   scripts/  (all scripts preserved)
>   sqlite-query skill (if installed)
> ```

The v2.2→v2.3 migration declares `scheduled-tasks/ops/` (all ops files) fully UNTOUCHED.

**Criteria**
The v2.2→v2.3 migration must leave the user's installation operationally equivalent to a fresh v2.3 setup. AUD-2026-04-25-018 fixed `template/scheduled-tasks/ops/conventions.md` — changing the Filing Answers section heading from `(Query Step 2 / Step 3)` to `(Query Step 1 / Step 2)` and updating the body text from "After any Step 2 (wiki) or Step 3 (web) answer…" to "After any Step 1 (wiki synthesis) or Step 2 (web) answer…". A user on v2.2 has the old conventions.md with the stale step numbers. The migration must re-copy this file to complete the v2.3 upgrade.

**Cause**
AUD-018 updated `template/scheduled-tasks/ops/conventions.md` after the v2.2→v2.3 migration path was written. The migration path was not updated to re-copy the file. Same root class as Finding 023: AUD-004 fixes were not cascaded to migrate.md.

**Consequence**
A v2.2 user who runs `!! migrate` will have a `CLAUDE.md` using v2.3 step numbers (Step 1 = wiki, Step 2 = web, Step 3 = training) but a `conventions.md` whose Filing Answers section still reads "After any Step 2 (wiki) or Step 3 (web) answer, ask: 'Worth filing this as an analysis page?'" Under the v2.3 numbering, this instruction activates only after web (Step 2) and training (Step 3) answers — wiki answers (now Step 1) will never trigger the filing offer. This is the exact behavioral regression AUD-018 was designed to prevent, now reintroduced via the migration path.

**Recommendation**
In the v2.2→v2.3 path:

1. Add a step (after step c) to re-copy `conventions.md`:
```
c2. Copy `blueprint/template/scheduled-tasks/ops/conventions.md` → `scheduled-tasks/ops/conventions.md`
```

2. Update the approval request to reflect the new file update:
```
Files UPDATED:
  CLAUDE.md  (Query Routing Rule section replaced — behavior change, no size change)
  scheduled-tasks/ops/conventions.md  (Filing Answers step labels updated to v2.3 numbering)

Files UNTOUCHED:
  wiki/  (all pages, index, log, hot preserved)
  memory.md, raw/, drafts/
  scheduled-tasks/ops/  (all other ops files preserved)
  scripts/  (all scripts preserved)
  sqlite-query skill (if installed)
```

### [AUD-2026-04-25-025] `blueprint-sync.md` missing `migrate.md` as cascade target for routing-rule changes

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-025 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/blueprint-sync.md` |
| **Location** | Cascade table |
| **Owner** | — |
| **Target Date** | — |

**Condition**
The cascade table has no row covering `migrate.md` as a target when the Query Routing Rule or ops files change.

> The row `Schema or startup change` lists: `blueprint/README.md`, `blueprint/setup-guide.md`, `blueprint/user-guide.md`, `blueprint/template/CLAUDE.md`. No mention of `migrate.md`.
> The row `Operation step change` lists: `blueprint/user-guide.md`, `blueprint/template/CLAUDE.md`, `blueprint/template/scheduled-tasks/ops/[op].md`. No mention of `migrate.md`.

**Criteria**
`migrate.md` embeds verbatim replacement blocks for schema-critical content: the entire `## Query Routing Rule` section (v2.2→v2.3 step c) and ops-file copies (v2.1→v2.3 step g). Any change to that content in `template/CLAUDE.md` or the relevant ops files must also propagate to the corresponding replacement blocks in `migrate.md`. Without a cascade rule, this propagation is invisible to the Blueprint Sync Rule and is silently skipped.

**Cause**
`migrate.md` was introduced in v2.2 after the cascade table was established. The table was never extended to cover `migrate.md`'s embedded blocks. Findings AUD-023 and AUD-024 are the direct consequence: two AUD-004 fixes applied to template files were not cascaded to `migrate.md` because the cascade table provided no guidance to do so.

**Consequence**
Any future change to the Query Routing Rule in `template/CLAUDE.md`, or to any ops file copied by the v2.1→v2.3 migration, will again miss `migrate.md`. The same CRITICAL-class divergence between fresh-setup and migrated users will recur. This finding is not a one-off mistake — it is a structural gap that will generate repeated CRITICALs until the cascade table is extended.

**Recommendation**
Add the following row to the cascade table in `blueprint-sync.md`:

```
| Query Routing Rule change | `blueprint/template/CLAUDE.md` (always) + `blueprint/template/scheduled-tasks/ops/migrate.md` (update the step c replacement block in the active migration path to match the new rule) |
```

Also add a general row for ops-file changes that are part of migration copy lists:

```
| Ops file change that is copied by the v2.1→v2.3 migration step g | `blueprint/template/scheduled-tasks/ops/migrate.md` (verify the updated file is listed in step g; add it if absent) |
```

### [AUD-2026-04-25-026] `audit.md` estimate command and range omit `CHANGELOG.md` after AUD-021

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-026 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/audit.md` |
| **Location** | Notes section — estimate command and token range |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> ```
> For `!! audit all`, expect ~30,000–47,000 tokens of reads for the tracked files. Run `python scripts/estimate_tokens.py blueprint/README.md blueprint/setup-guide.md blueprint/user-guide.md blueprint/troubleshooting.md blueprint/template/CLAUDE.md blueprint/template/scheduled-tasks/refresh-hot.md blueprint/template/scheduled-tasks/ops/*.md blueprint/skills/sqlite-query/*.md blueprint/skills/claude-code-enhanced/*.md` for a live estimate. Warn the user up front if the session is already close to context limits.
> ```

`blueprint/CHANGELOG.md` is absent from the estimate command. The stated range (~30,000–47,000) was calibrated without CHANGELOG.md.

**Criteria**
The estimate command must cover all files in the `!! audit all` scope. AUD-021 added `blueprint/CHANGELOG.md` to the scope (now present at audit.md line 19). The estimate command must be updated to include it, and the expected range must be recalibrated to account for CHANGELOG.md's actual size (~30,500 tokens per read-tool estimate on this date).

**Cause**
AUD-021's fix updated the scope list but not the estimate command or the range in the same Notes block. The two items in the Notes section are coupled — adding a file to scope without updating the estimate creates a false-precision estimate for the largest single file in the corpus.

**Consequence**
An agent running the estimate command before `!! audit all` will receive a figure of approximately 35,000 tokens (the pre-CHANGELOG scope) and may proceed without issuing the context-limit warning. The actual read cost is approximately 65,000+ tokens — nearly double the upper bound stated. On a session that already has significant context, this can exhaust the context window mid-audit with no prior warning. The session-advisory and context-limit protection that `audit.md` is designed to provide are silently bypassed.

**Recommendation**
1. Add `blueprint/CHANGELOG.md` to the estimate command:

```
python scripts/estimate_tokens.py blueprint/README.md blueprint/setup-guide.md blueprint/user-guide.md blueprint/troubleshooting.md blueprint/CHANGELOG.md blueprint/template/CLAUDE.md blueprint/template/scheduled-tasks/refresh-hot.md blueprint/template/scheduled-tasks/ops/*.md blueprint/skills/sqlite-query/*.md blueprint/skills/claude-code-enhanced/*.md
```

2. Update the expected range. Prior scope total was ~34,819 tokens (AUD-004 appendix). CHANGELOG.md adds ~30,500 tokens. New expected range: **~60,000–70,000 tokens**. Update the Notes line to:

```
For `!! audit all`, expect ~60,000–70,000 tokens of reads for the tracked files (CHANGELOG.md alone accounts for ~30,000+ tokens and grows with every audit cycle).
```
