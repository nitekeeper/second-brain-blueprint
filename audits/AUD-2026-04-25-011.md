### [AUD-2026-04-25-042] `query-layer.md` and `conventions.md` use Unix-only `find` command, silently disabling sqlite-query on Windows

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-042 |
| **Severity** | WARNING |
| **Status** | RESOLVED |
| **File** | `skills/sqlite-query/query-layer.md`, `template/scheduled-tasks/ops/conventions.md` |
| **Location** | query-layer.md Step 1 Python block (slug-to-path resolution); conventions.md Query Layer Hook Contract |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `result = subprocess.run(`
> `    ["find", pages_dir, "-name", f"{row[0]}.md"],`
> `    capture_output=True, text=True`
> `)`
> `path = result.stdout.strip()`
> `if path:`
> `    candidate_paths.append(path)`

And in `conventions.md` Query Layer Hook Contract:
> `Use \`find wiki/pages -name "<slug>.md"\` (via \`subprocess.run\`) to resolve slugs to concrete paths.`

Both files use the Unix `find` utility to resolve page slugs to file paths. On Windows, `subprocess.run(["find", pages_dir, "-name", slug])` invokes `C:\Windows\System32\find.exe`, which is a text-search utility with incompatible syntax. It receives the args as a search string and filename, produces no matching output, and `result.stdout.strip()` returns `""` — so no path is added to `candidate_paths`. After the loop, `candidate_paths` is empty, which triggers the fallback branch in Step 2 (`"If candidate_paths is empty (topic not in DB): fall back to grep"`).

**Criteria**
The blueprint explicitly documents cross-platform support across Windows, macOS, and Linux (troubleshooting.md: "The agent uses a set of Python scripts under `scripts/` for file operations that previously required bash. These scripts work identically on Windows, macOS, and Linux." and "Python 3.8+ is required."). The sqlite-query skill's install flow (SKILL.md) is offered to Claude Code CLI users — who may be on Windows. Cross-platform path resolution must use Python's `pathlib` or equivalent, not OS-specific shell utilities. The conventions.md Hook Contract is the authoritative guidance for future skill authors — a bad pattern here propagates to all future query-layer skill implementations.

**Cause**
`query-layer.md` and the conventions.md Hook Contract were written assuming a POSIX environment (Cowork Bash sandbox runs on Linux). The general cross-platform requirement (applied in `scripts/*.py` via pathlib) was not applied to the skill's query-layer code. No Windows-specific test path exists in the blueprint's development process to catch this category of failure.

**Consequence**
On Windows Claude Code environments, the sqlite-query skill's query-layer always produces `candidate_paths = []`, unconditionally falling back to grep on every query. The ingest hook (pure Python sqlite3) continues to keep wiki.db populated correctly, so the DB accumulates data that is never queried. The degradation is completely silent — no error or warning is surfaced to the user. The user who ran the installation and backfill gets no benefit from either. Additionally, conventions.md's Hook Contract directs future skill authors to use the same broken pattern, meaning any third-party query-layer skill would have the same Windows incompatibility.

**Recommendation**

In `skills/sqlite-query/query-layer.md`, replace the `subprocess.run(["find", ...])` block with a cross-platform `pathlib.rglob()` call:

Change (in the Step 1 Python block):
```python
candidate_paths = []
pages_dir = str(WORKDIR / "wiki" / "pages")
for row in rows:
    result = subprocess.run(
        ["find", pages_dir, "-name", f"{row[0]}.md"],
        capture_output=True, text=True
    )
    path = result.stdout.strip()
    if path:
        candidate_paths.append(path)
```

To:
```python
candidate_paths = []
pages_dir = WORKDIR / "wiki" / "pages"
for row in rows:
    matches = list(pages_dir.rglob(f"{row[0]}.md"))
    if matches:
        candidate_paths.append(str(matches[0]))  # rglob returns Path objects
```

Remove the `import subprocess` if it is no longer needed elsewhere in the block.

In `template/scheduled-tasks/ops/conventions.md` Query Layer Hook Contract, replace:
```
Use `find wiki/pages -name "<slug>.md"` (via `subprocess.run`) to resolve slugs to concrete paths.
```
With:
```
Use `pathlib.rglob(f"{slug}.md")` on the `wiki/pages` directory (cross-platform — works on Windows, macOS, and Linux). Do NOT use `subprocess.run(["find", ...])` — it invokes OS-specific utilities that fail silently on Windows.
```

### [AUD-2026-04-25-043] `CLAUDE.md` Step 2 omits "Worth filing as analysis page?" prompt

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-043 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/CLAUDE.md` |
| **Location** | Query Routing Rule — Step 2 (Web Search) bullet list |
| **Owner** | — |
| **Target Date** | — |

**Condition**
Step 1 of the Query Routing Rule in CLAUDE.md contains:
> `5. Ask: "Worth filing as an analysis page?" — if yes, read \`@scheduled-tasks/ops/conventions.md\` first`

Step 2 contains only three bullets (search and summarize; ingest if directly answered; ingest + cite both sources if partial wiki hit; inform Step 3 if loosely related). No "Worth filing as analysis page?" prompt appears in Step 2.

**Criteria**
`conventions.md` (Filing Answers section) states: "After any Step 1 (wiki synthesis) or Step 2 (web) answer, ask: 'Worth filing this as an analysis page?'" `user-guide.md` (Asking Questions section) confirms: "After a Step 1 (wiki synthesis) or Step 2 (web) answer, the agent will ask: 'Worth filing this as an analysis page?'" Both of these files — the agent's convention rules and the user-facing guide — document this as a post-Step-2 behavior. CLAUDE.md is the authoritative instruction set the agent follows on every response; if it omits the prompt, agents will not reliably offer analysis-page filing after web search answers.

**Cause**
The "Worth filing?" prompt was part of the original Step 1 workflow and was encoded into CLAUDE.md's Step 1 numbered sub-steps. When Step 2's bullet format was written or revised, the mirrored prompt was not included. The conventions.md and user-guide.md separately document the expected behavior, but CLAUDE.md governs actual agent behavior.

**Consequence**
Agents following CLAUDE.md's Query Routing Rule strictly will not ask "Worth filing this as an analysis page?" after Step 2 web search answers. The user loses the opportunity to persist a web-derived answer as a permanent analysis page (without knowing why the prompt doesn't appear). The inconsistency produces different agent behavior depending on whether the answer came from the wiki (Step 1) or the web (Step 2) — diverging from the documented user-facing behavior. No data loss; purely a feature miss.

**Recommendation**

In `template/CLAUDE.md`, add a final bullet to Step 2 after the three condition bullets:

Change:
```
**Step 2 — Web Search**
Runs when: (a) wiki returned nothing useful, OR (b) question needs current or recent information.
- Search and summarize
- If the result **directly answered the question** → silently save to `wiki/inbox/` and read `@scheduled-tasks/ops/ingest.md` to ingest it
- If the result **directly answered the question AND Step 1 returned partial wiki content** → ingest the web result; answer citing both the wiki pages and the web result
- If the result is **loosely related but did not answer** → skip ingest; use partial findings to inform Step 3
```

To:
```
**Step 2 — Web Search**
Runs when: (a) wiki returned nothing useful, OR (b) question needs current or recent information.
- Search and summarize
- If the result **directly answered the question** → silently save to `wiki/inbox/` and read `@scheduled-tasks/ops/ingest.md` to ingest it
- If the result **directly answered the question AND Step 1 returned partial wiki content** → ingest the web result; answer citing both the wiki pages and the web result
- If the result is **loosely related but did not answer** → skip ingest; use partial findings to inform Step 3
- After summarizing: ask "Worth filing this as an analysis page?" — if yes, read `@scheduled-tasks/ops/conventions.md` first
```

### [AUD-2026-04-25-044] `reference.md` blueprint/ tree omits `docs/superpowers/` directory

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-044 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/reference.md` |
| **Location** | Directory Structure section — blueprint/ subtree, docs/ entry |
| **Owner** | — |
| **Target Date** | — |

**Condition**
The blueprint/ subtree in `reference.md` shows:
> `├── docs/                   ← Audit report template and design specs. Developer use only.`
> `│   └── audit-report-template.md`

The blueprint repo contains a tracked `docs/superpowers/` directory with implementation plans and design specs (`docs/superpowers/plans/` and `docs/superpowers/specs/`). This directory is absent from the reference.md diagram.

**Criteria**
`reference.md` is the canonical directory map read by the agent when answering directory structure questions (per CLAUDE.md Ops Routing: "Directory / structure query → `@scheduled-tasks/ops/reference.md`"). The diagram should accurately represent the blueprint repo contents so that agents and maintainers working in blueprint-authoring mode can correctly answer layout questions.

**Cause**
`docs/superpowers/` was added as a developer workfiles directory for implementation plans and design specs, likely during the initial development of the audit-driven workflow. The reference.md diagram was not updated to include it; only `audit-report-template.md` (added by AUD-037 in a prior cycle) is listed.

**Consequence**
An agent asked about the directory structure in blueprint-authoring mode would report that `docs/` contains only `audit-report-template.md`. A blueprint maintainer asking "where are the implementation plans stored?" would get an incorrect answer from reference.md. The `docs/superpowers/` directory is developer-only and does not affect deployed users; the impact is limited to blueprint-maintainer queries. The failure mode is documentation drift, not functional breakage.

**Recommendation**

In `template/scheduled-tasks/ops/reference.md`, expand the `docs/` subtree in the blueprint/ section:

Change:
```
│   ├── docs/                   ← Audit report template and design specs. Developer use only.
│   │   └── audit-report-template.md
```

To:
```
│   ├── docs/                   ← Audit report template and developer workfiles. Maintainer use only.
│   │   ├── audit-report-template.md
│   │   └── superpowers/        ← Implementation plans and design specs (not distributed to end users)
│   │       ├── plans/
│   │       └── specs/
```

### [AUD-2026-04-25-045] `refresh-hot.md` Step 3 does not detect `claude-code-enhanced` skill

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-045 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/refresh-hot.md` |
| **Location** | Step 3 — Derive `Active skills` |
| **Owner** | — |
| **Target Date** | — |

**Condition**
Step 3 of `refresh-hot.md` reads:
> `Check whether \`scheduled-tasks/query-layer.md\` exists (\`python scripts/file_check.py scheduled-tasks/query-layer.md\`). If it exists, the \`sqlite-query\` skill is installed — emit \`sqlite-query\`. If absent, emit \`none\`. (As additional skills are added, this step expands to check their respective hook files and append their names to the list.)`

The `claude-code-enhanced` skill (a bundled skill in `skills/claude-code-enhanced/`) installs `scheduled-tasks/claude-code-enhanced.md` when activated. Step 3 only checks for `query-layer.md` and has no check for `claude-code-enhanced.md`.

**Criteria**
The `Active skills:` field in `hot.md` is read at every session startup to orient the agent about the installed capability set. If `claude-code-enhanced` is installed, `Active skills:` should reflect that. The parenthetical "(As additional skills are added, this step expands…)" acknowledges the gap pattern but does not address the fact that `claude-code-enhanced` is already a first-party bundled skill in the current blueprint — not a hypothetical future addition.

**Cause**
`refresh-hot.md` Step 3 was written when only the `sqlite-query` skill existed. `claude-code-enhanced` was added later with its own hook file (`scheduled-tasks/claude-code-enhanced.md`), and Step 3 was not updated to detect it. The parenthetical was added as a forward-looking note but did not trigger the required backward-looking update.

**Consequence**
When `claude-code-enhanced` is installed and `sqlite-query` is not, `Active skills:` shows `none`. When both are installed, `Active skills:` shows only `sqlite-query`. The agent's startup orientation (read from hot.md) is inaccurate about the installed skill set. Functionally, slash commands `/wrap`, `/ready`, and `/migrate` work regardless of what hot.md says (they invoke the skill via `scheduled-tasks/claude-code-enhanced.md` directly); the consequence is orientation inaccuracy, not broken behavior. Documentation-only impact.

**Recommendation**

In `template/scheduled-tasks/refresh-hot.md`, expand Step 3 to also check for `claude-code-enhanced.md`:

Change:
```
3. **Derive `Active skills`:** Check whether `scheduled-tasks/query-layer.md` exists (`python scripts/file_check.py scheduled-tasks/query-layer.md`). If it exists, the `sqlite-query` skill is installed — emit `sqlite-query`. If absent, emit `none`. (As additional skills are added, this step expands to check their respective hook files and append their names to the list.)
```

To:
```
3. **Derive `Active skills`:** Check each skill's hook or installed file:
   - `python scripts/file_check.py scheduled-tasks/query-layer.md` — if present, add `sqlite-query` to the list
   - `python scripts/file_check.py scheduled-tasks/claude-code-enhanced.md` — if present, add `claude-code-enhanced` to the list

   If neither is present, emit `none`. If one or more are present, emit them comma-separated (e.g. `sqlite-query, claude-code-enhanced`). When a new skill is added to the blueprint, add its detection check here at the same time.
```
