### [AUD-2026-04-26-046] `migrate.md` v2.2‚Üív2.3 step c missing Step 2 "Worth filing?" bullet

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-046 |
| **Severity** | WARNING |
| **Status** | OPEN |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.2 ‚Üí v2.3 Path ‚Äî step c replacement block, Step 2 bullet list |
| **Owner** | ‚Äî |
| **Target Date** | ‚Äî |

**Condition**
> ```
> **Step 2 ‚Äî Web Search**
> Runs when: (a) wiki returned nothing useful, OR (b) question needs current or recent information.
> - Search and summarize
> - If the result **directly answered the question** ‚Üí silently save to `wiki/inbox/` and read `@scheduled-tasks/ops/ingest.md` to ingest it
> - If the result **directly answered the question AND Step 1 returned partial wiki content** ‚Üí ingest the web result; answer citing both the wiki pages and the web result
> - If the result is **loosely related but did not answer** ‚Üí skip ingest; use partial findings to inform Step 3
> ```

The inline Query Routing Rule replacement block (v2.2‚Üív2.3, step c) ends Step 2 after the three condition bullets. The current `template/CLAUDE.md` Step 2 contains a fourth bullet:
> `- After summarizing: ask "Worth filing this as an analysis page?" ‚Äî if yes, read \`@scheduled-tasks/ops/conventions.md\` first`

This bullet is absent from the migrate.md replacement block.

**Criteria**
`blueprint-sync.md`'s cascade table includes the row: "Query Routing Rule change ‚Üí `blueprint/template/CLAUDE.md` (always) + `blueprint/template/scheduled-tasks/ops/migrate.md` (update the step c replacement block in the active migration path to match the new rule)." AUD-043 was a Query Routing Rule change (Step 2 bullet added to `CLAUDE.md`). The documented cascade required updating the migrate.md step c block. This requirement was not fulfilled when AUD-043 was implemented ‚Äî the AUD-043 design spec's "Files Affected" table listed only `template/CLAUDE.md`, omitting `migrate.md`.

**Cause**
The AUD-043 fix was applied to `template/CLAUDE.md` directly; the design spec did not include `migrate.md` in its affected-files list, and the cascade rule was not checked at fix time. The v2.1‚Üív2.3 migration path is unaffected because step c in that path copies `template/CLAUDE.md` wholesale (all fixes propagate automatically). Only the v2.2‚Üív2.3 path is vulnerable ‚Äî it uses an inline replacement block that must be kept manually in sync with `CLAUDE.md`.

**Consequence**
Any user running `!! migrate` from v2.2 to v2.3 receives a CLAUDE.md whose Step 2 is missing the "Worth filing?" prompt. The migrated agent will not ask "Worth filing this as an analysis page?" after web search answers ‚Äî diverging from the behavior documented in `user-guide.md` ("After a Step 1 or Step 2 answer, the agent will ask: 'Worth filing this as an analysis page?'") and `conventions.md`. This is the same silent behavioral regression that AUD-043 originally fixed, re-introduced on every v2.2‚Üív2.3 migration. The failure is invisible to the user ‚Äî no error, no warning, just missing prompts.

**Recommendation**

In `template/scheduled-tasks/ops/migrate.md`, add the missing fourth bullet to the Step 2 block inside the v2.2‚Üív2.3 step c replacement text:

Change:
```
- If the result is **loosely related but did not answer** ‚Üí skip ingest; use partial findings to inform Step 3
```

To:
```
- If the result is **loosely related but did not answer** ‚Üí skip ingest; use partial findings to inform Step 3
- After summarizing: ask "Worth filing this as an analysis page?" ‚Äî if yes, read `@scheduled-tasks/ops/conventions.md` first
```

(The change is a single line appended to the Step 2 bullet list, immediately before the `**Step 3 ‚Äî Training Knowledge**` heading.)

### [AUD-2026-04-26-047] Migration paths do not propagate AUD-045 `refresh-hot.md` fix; cascade table has no coverage

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-047 |
| **Severity** | STYLE |
| **Status** | OPEN |
| **File** | `template/scheduled-tasks/ops/migrate.md`, `template/scheduled-tasks/ops/blueprint-sync.md` |
| **Location** | migrate.md v2.2‚Üív2.3 approval request (Files UPDATED); blueprint-sync.md Refresh-hot.md cascade row |
| **Owner** | ‚Äî |
| **Target Date** | ‚Äî |

**Condition**
The v2.2‚Üív2.3 approval request lists:
> ```
> Files UPDATED:
>   CLAUDE.md  (Query Routing Rule section replaced ‚Äî behavior change, no size change)
>   scheduled-tasks/ops/conventions.md  (Filing Answers step labels updated to v2.3 numbering)
>   scheduled-tasks/ops/blueprint-sync.md  (two new v2.3 cascade rows added)
>   wiki/hot.md  (Schema: v2.2 ‚Üí v2.3)
> ```

`scheduled-tasks/refresh-hot.md` is absent from this list. AUD-045 updated `template/scheduled-tasks/refresh-hot.md` to detect `claude-code-enhanced` in Step 3. The v2.2‚Üív2.3 migration does not copy the updated template file to `scheduled-tasks/refresh-hot.md`, leaving migrated wikis with a pre-AUD-045 version.

Additionally, `blueprint-sync.md`'s Refresh-hot.md cascade row reads:
> ```
> | Refresh-hot.md change | `blueprint/template/scheduled-tasks/refresh-hot.md`, `blueprint/template/CLAUDE.md` (hot.md Format block), `blueprint/setup-guide.md` (initial hot.md snippet) |
> ```

`migrate.md` is not listed. There is no documented cascade requirement that triggers a migrate.md update when `refresh-hot.md` changes ‚Äî the gap in the cascade table is the root cause.

The v2.1‚Üív2.3 migration path has the same gap: step g copies `audit.md`, `ingest.md`, `lint.md`, `conventions.md` from `blueprint/template/scheduled-tasks/ops/` ‚Äî it does not copy `refresh-hot.md` (which is in `blueprint/template/scheduled-tasks/`, one level up from `ops/`).

**Criteria**
`refresh-hot.md` is a deployed file that participates in every wiki state change. When the template is updated to fix a detection gap (AUD-045), existing deployed wikis should receive the fix via the migration path. Without a cascade rule covering this case, `refresh-hot.md` improvements are silently dropped for all users who do `!! migrate` rather than a fresh setup.

**Cause**
The blueprint-sync.md cascade table was not updated when AUD-045 fixed `refresh-hot.md`. The "Refresh-hot.md change" row predates the migration op and was never amended to include a "verify migrate.md propagates this" clause. The v2.2‚Üív2.3 migration approval message was also not updated to include `refresh-hot.md` when AUD-045 was applied.

**Consequence**
Users who migrate from v2.2 (or v2.1) to v2.3 end up with a `scheduled-tasks/refresh-hot.md` that only detects `sqlite-query` (the pre-AUD-045 version). If `claude-code-enhanced` is installed, `Active skills:` in `hot.md` will show `sqlite-query` (if installed) or `none` ‚Äî never `claude-code-enhanced`. The agent's startup orientation is inaccurate. No functional breakage; the skill itself works via its installed files regardless of what `hot.md` says.

**Recommendation**

Two changes:

1. In `template/scheduled-tasks/ops/migrate.md`, add `scheduled-tasks/refresh-hot.md` to the v2.2‚Üív2.3 approval request (Files UPDATED) and add the corresponding step to copy it:

   In the approval request block, add:
   ```
     scheduled-tasks/refresh-hot.md  (Step 3 updated to detect claude-code-enhanced skill)
   ```

   After step c3 (copy blueprint-sync.md), add a new step:
   ```
   c4. Copy `blueprint/template/scheduled-tasks/refresh-hot.md` ‚Üí `scheduled-tasks/refresh-hot.md`
   ```

   For the v2.1‚Üív2.3 path: Add `refresh-hot.md` to the Files UPDATED list, and add a corresponding copy step (parallel to step g but targeting the `scheduled-tasks/` parent, not `scheduled-tasks/ops/`).

2. In `template/scheduled-tasks/ops/blueprint-sync.md`, update the Refresh-hot.md cascade row to include migrate.md:

   Change:
   ```
   | Refresh-hot.md change | `blueprint/template/scheduled-tasks/refresh-hot.md`, `blueprint/template/CLAUDE.md` (hot.md Format block), `blueprint/setup-guide.md` (initial hot.md snippet) |
   ```

   To:
   ```
   | Refresh-hot.md change | `blueprint/template/scheduled-tasks/refresh-hot.md`, `blueprint/template/CLAUDE.md` (hot.md Format block), `blueprint/setup-guide.md` (initial hot.md snippet), `blueprint/template/scheduled-tasks/ops/migrate.md` (add copy step for both migration paths; update Files UPDATED lists) |
   ```

### [AUD-2026-04-26-048] Cold-start token estimate (~1,080) has drifted ~2√ó from current `estimate_tokens.py` reading

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-048 |
| **Severity** | STYLE |
| **Status** | OPEN |
| **File** | `README.md`, `user-guide.md` |
| **Location** | README.md "Key Features" bullet; user-guide.md intro paragraph and "Token Awareness" table |
| **Owner** | ‚Äî |
| **Target Date** | ‚Äî |

**Condition**
`README.md` states:
> `**Lean startup** ‚Äî agent cold-starts in ~1,080 tokens; reads only what it needs`

`user-guide.md` states:
> `Total cold-start cost: ~1,080 tokens. This is intentionally lean.`

and in the Token Awareness table:
> `| Cold start | ~1,080 |`

`template/CLAUDE.md` is now **~2,013 tokens** per `python template/scripts/estimate_tokens.py` (chars √∑ 4 ‚Äî the blueprint's own measurement standard). The deployed CLAUDE.md (template minus the `> **Setup note:**` block ‚âà ~1,990 tokens) plus `wiki/hot.md` (~80 tokens) puts the current cold-start estimate at approximately **~2,070 tokens** ‚Äî roughly 2√ó the stated ~1,080.

**Criteria**
`audit.md` notes explicitly list "documented token estimates can drift from reality" as a finding category for this codebase. `user-guide.md` itself states: "Note: All token numbers below...are estimates derived from chars √∑ 4." The chars √∑ 4 metric is the authoritative measurement standard the blueprint uses for all estimates. Even by this standard, the stated cold-start figure is approximately 2√ó too low.

**Cause**
The ~1,080 figure was accurate at the time of the v2.1‚Üív2.3 schema redesign, which was specifically aimed at reducing CLAUDE.md from ~7,780 tokens to ~1,080. Since v2.3, CLAUDE.md has grown through successive audit cycles (Query Routing Rule additions, Ops Routing table rows, Core Rules clarifications). AUD-011 already showed `template/CLAUDE.md` at ~1,450 tokens; the current reading is ~2,013 tokens. The cold-start estimate was never updated after the v2.3 baseline. The user-guide.md note about estimates varying "sometimes noticeably" acknowledges drift but does not cover a 2√ó divergence from the blueprint's own chars√∑4 tool.

**Consequence**
Users planning session budgets around a ~1,080-token cold-start will find their sessions running ~1,000 tokens heavier per startup than expected. In a 200,000-token context window, the discrepancy is not catastrophic, but it undermines the accuracy of the blueprint's own token-awareness guidance. The "Lean startup" feature highlight in README.md overstates the benefit by approximately 2√ó.

**Recommendation**

Update the cold-start figures in both files to reflect the current `estimate_tokens.py` reading (~2,070 tokens):

In `README.md`, change:
```
**Lean startup** ‚Äî agent cold-starts in ~1,080 tokens; reads only what it needs
```
To:
```
**Lean startup** ‚Äî agent cold-starts in ~2,000 tokens; reads only what it needs
```

In `user-guide.md`, change the intro paragraph and Token Awareness table entry from ~1,080 to ~2,000. Also update the `!! ready` cold-start figure ("~1,830 tokens total" ‚Üí recalibrate similarly: `!! ready` = cold-start + session-memory read, so ~2,000 + ~750 session-memory ‚âà ~2,750 tokens total if memory is full ‚Äî run `estimate_tokens.py` on `memory.md` at a representative size to compute the correct figure).

> **Note:** The migration confirmation message in `migrate.md` ("Cold-start: ~7,780 ‚Üí ~1,080 tokens") is a historical statement about the v2.1‚Üív2.3 improvement; it need not be updated since it describes the state at migration time. Only the forward-looking user-guide.md and README.md claims need correction.

### [AUD-2026-04-26-049] `audit-report-template.md` lists unreachable `MEDIUM` risk level with no derivation rule

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-26-049 |
| **Severity** | STYLE |
| **Status** | OPEN |
| **File** | `docs/audit-report-template.md` |
| **Location** | Executive Summary section ‚Äî Overall Risk line and derivation note |
| **Owner** | ‚Äî |
| **Target Date** | ‚Äî |

**Condition**
The template's Executive Summary section shows:
> `**Overall Risk:** \`CRITICAL\` / \`HIGH\` / \`MEDIUM\` / \`LOW\` / \`CLEAN\``

Immediately below, the derivation note reads:
> `Risk is derived from the highest-severity open finding in priority order: (1) any unresolved CRITICAL ‚Üí CRITICAL; (2) open WARNINGs (no open CRITICALs) ‚Üí HIGH; (3) only resolved findings and/or open STYLEs ‚Üí LOW; (4) zero findings ‚Üí CLEAN. STYLE findings do not elevate risk above LOW ‚Äî they are documentation-only.`

The derivation maps exactly four states: CRITICAL, HIGH, LOW, CLEAN. There is no path to MEDIUM.

**Criteria**
A template field that lists an option with no corresponding selection rule creates ambiguity for anyone filling in the template ‚Äî it implies MEDIUM is a valid state that might be appropriate in some unstated condition. `audit.md`'s own derivation (Step 4c) also omits MEDIUM: "CRITICAL if any open CRITICALs ‚Üí `CRITICAL`; open WARNINGs only ‚Üí `HIGH`; open STYLEs only ‚Üí `LOW`; zero open findings ‚Üí `CLEAN`." The two sources agree: MEDIUM is not part of the risk vocabulary. Its presence in the template option list is a stale artifact.

**Cause**
MEDIUM was likely included in the template's risk options as a placeholder during initial authoring (covering a hypothetical severity tier between HIGH and LOW). The risk derivation rules were subsequently tightened to the current four-state model, but the option list on the Overall Risk line was not updated to match.

**Consequence**
Any future agent filling in the template might select MEDIUM when finding a borderline situation with no clear HIGH or LOW mapping. This produces an inconsistent audit report whose risk level falls outside the defined vocabulary ‚Äî a subtle documentation inconsistency that compounds if MEDIUM appears in an action-item checklist or executive summary. No functional impact on existing reports (all 11 prior reports correctly used CRITICAL, HIGH, LOW, or CLEAN).

**Recommendation**

In `docs/audit-report-template.md`, remove `MEDIUM` from the Overall Risk line:

Change:
```
**Overall Risk:** `CRITICAL` / `HIGH` / `MEDIUM` / `LOW` / `CLEAN`
```

To:
```
**Overall Risk:** `CRITICAL` / `HIGH` / `LOW` / `CLEAN`
```

No change to the derivation note ‚Äî it is already correct and does not reference MEDIUM.
