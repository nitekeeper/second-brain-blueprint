### [AUD-2026-04-25-030] `migrate.md` v2.2в†’v2.3 path does not update `blueprint-sync.md` to v2.3 rows

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-030 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | v2.2в†’v2.3 path вЂ” approval request "Files UNTOUCHED" block and execution steps |
| **Owner** | вЂ” |
| **Target Date** | вЂ” |

**Condition**
> `Files UNTOUCHED:`
> `  scheduled-tasks/ops/  (all other ops files preserved)`

The v2.2в†’v2.3 execution steps contain no instruction to copy `blueprint-sync.md` from the template. The approval request's UNTOUCHED block confirms this by implication: `blueprint-sync.md` is among the "all other ops files preserved."

**Criteria**
After migration, the user's `scheduled-tasks/ops/` environment should match the v2.3 template. The current v2.3 `template/scheduled-tasks/ops/blueprint-sync.md` contains two rows added as part of the v2.3 work (AUD-025 fix) that are absent from the v2.2 version: `Query Routing Rule change` (instructs maintaining `migrate.md`'s step c block when the routing rule changes) and `Ops file change that is copied by the v2.1в†’v2.3 migration step g` (instructs updating `migrate.md`'s step g list). A v2.2 user who runs `!! migrate` does not receive these rows.

**Cause**
The v2.2в†’v2.3 path was designed as a minimal behavioral patch (routing rule section swap + conventions.md step labels). `blueprint-sync.md` contains maintainer-facing cascade rules вЂ” not direct behavioral instructions вЂ” so it was not included in the patch's target list. AUD-025's fix updated the template but the migration path was not revised to propagate the change.

**Consequence**
A user who migrates from v2.2 to v2.3 via `!! migrate` and subsequently acts as a blueprint maintainer (editing `template/CLAUDE.md`'s Query Routing Rule) will read their deployed `blueprint-sync.md` and not find the `Query Routing Rule change` row. They will not be reminded to update `migrate.md`'s step c block to match the new rule вЂ” silently breaking future migrations. Impact is limited to users who are both (a) migrating from v2.2 (not a fresh v2.3 install) and (b) maintaining the blueprint template.

**Recommendation**
Add a step to the v2.2в†’v2.3 execution path:

```
c3. Copy `blueprint/template/scheduled-tasks/ops/blueprint-sync.md` в†’ `scheduled-tasks/ops/blueprint-sync.md`
```

And add one line to the Files UPDATED block in the approval request:

```
scheduled-tasks/ops/blueprint-sync.md  (two new v2.3 cascade rows added)
```

Remove `blueprint-sync.md` from the implicit "all other ops files preserved" umbrella in the UNTOUCHED block by noting it explicitly in UPDATED instead.

### [AUD-2026-04-25-031] `reference.md` blueprint/ directory tree omits `ROADMAP.md`

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-031 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/reference.md` |
| **Location** | blueprint/ section of the directory tree |
| **Owner** | вЂ” |
| **Target Date** | вЂ” |

**Condition**
The `blueprint/` node in `reference.md`'s directory tree lists: `LICENSE`, `.gitignore`, `README.md`, `setup-guide.md`, `user-guide.md`, `troubleshooting.md`, `CHANGELOG.md`, `template/`, `skills/`. It does not list `ROADMAP.md`, which exists at `blueprint/ROADMAP.md` (verified present, not gitignored).

**Criteria**
The reference directory tree is the canonical layout guide for maintainers. `ROADMAP.md` is a tracked, first-class blueprint document: it is referenced in `blueprint-sync.md`'s `New skill bundle added` row (`blueprint/ROADMAP.md`) as a required cascade target when a new skill is shipped. A maintainer consulting `reference.md` to understand what `blueprint/` contains will not see `ROADMAP.md` and may overlook it when auditing or updating the blueprint.

**Cause**
`ROADMAP.md` was added to the blueprint but the `reference.md` blueprint directory tree was not updated. Same root cause and class as AUD-2026-04-25-022 (which omitted `template/scripts/` from the tree вЂ” now resolved).

**Consequence**
Low operational risk: `setup-guide.md` does not copy `ROADMAP.md` into the user's working folder, and regular wiki users never interact with it. However, a maintainer consulting `reference.md` will have an incomplete picture of what `blueprint/` contains. The `blueprint-sync.md` cascade rule for `New skill bundle added` references a file that the layout guide does not acknowledge.

**Recommendation**
Add `ROADMAP.md` to the `blueprint/` node in the directory tree, between `CHANGELOG.md` and `template/`:
