### [AUD-2026-04-25-034] `query-layer.md` step 3 fallback cross-reference is ambiguous

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-034 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `skills/sqlite-query/query-layer.md` |
| **Location** | Step 3 — last sentence |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `If grep also returns nothing, proceed to Step 3 of the core query op (read `index.md`).`

The referenced "Step 3 of the core query op" (CLAUDE.md Query Routing Rule, Step 1 sub-step 3) reads: "Grep `wiki/pages` for topic slug; if no match, read `wiki/index.md`." It includes a grep and an index read — not just the index read.

**Criteria**
The query-layer.md has already executed the grep in its own step 3 and found nothing. An agent following "proceed to Step 3 of the core query op" literally would re-run the same grep. The parenthetical "(read `index.md`)" is intended to signal "skip the grep part of step 3," but this intent is implicit and relies on the agent inferring that the grep was already done. Additionally, CLAUDE.md sub-step 2 says "fall back to step 3 on empty/failure" when the query layer returns empty, which is the same step 3 — so an agent exiting the query layer with no candidate paths would trigger the CLAUDE.md fallback and re-run grep anyway, independent of the query-layer.md instruction.

**Cause**
The query-layer.md tries to handle the full grep fallback internally (for directness) while still needing to hand off to CLAUDE.md's index-read step. The cross-reference was written to short-circuit the redundant grep by implying "(read `index.md`)" is all that remains, but it references a step that includes the grep as its first action.

**Consequence**
No functional failure — the worst case is that the agent runs the grep twice (once inside query-layer.md step 3, once when falling through to CLAUDE.md step 3). The double-grep wastes a small number of tokens but produces a correct result. The parenthetical "(read `index.md`)" is adequate guidance for a capable agent to infer the intent, but it is not unambiguous instruction.

**Recommendation**
Replace the ambiguous cross-reference with a direct instruction that eliminates the need to infer:

Change:
```
If grep also returns nothing, proceed to Step 3 of the core query op (read `index.md`).
```

To:
```
If grep also returns nothing, read `wiki/index.md` directly and continue from sub-step 4 of the query waterfall (read candidate pages; answer with [[wiki link]] citations).
```

This removes the cross-reference entirely and specifies the exact action, eliminating any risk of the agent re-running the grep.

### [AUD-2026-04-25-035] `migrate.md` v2.2→v2.3 rollback omits `blueprint-sync.md` revert

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-035 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/migrate.md` |
| **Location** | Rollback section — v2.2→v2.3 bullet |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `v2.2→v2.3: Rename backups/CLAUDE.md-v2.2-<date>.bak → CLAUDE.md; rename backups/conventions.md-v2.2-<date>.bak → scheduled-tasks/ops/conventions.md; revert Schema: in hot.md to v2.2.`

The rollback lists three steps: restore CLAUDE.md, restore conventions.md, revert hot.md. No mention of reverting `scheduled-tasks/ops/blueprint-sync.md`.

**Criteria**
The approval request for the v2.2→v2.3 migration explicitly lists `scheduled-tasks/ops/blueprint-sync.md` under "Files UPDATED" and the migration step c3 copies a new version from `blueprint/template/scheduled-tasks/ops/blueprint-sync.md`. However, no backup of `blueprint-sync.md` is taken (backup steps b and b2 cover only CLAUDE.md and conventions.md), and the Rollback section does not include restoring it. A complete rollback of all mutated files should include all three updated ops files, not just two.

**Cause**
The migration's backup steps were written to capture the two files with behavioral impact on the agent (CLAUDE.md contains the changed Query Routing Rule; conventions.md contains changed step labels). `blueprint-sync.md` was added to the migration to update its cascade guidance rows — a developer-tool change with no agent-behavior impact — and its backup was omitted as low-consequence.

**Consequence**
Documentation-only impact. A developer who rolls back the v2.2→v2.3 migration following the Rollback instructions will end up running v2.2 CLAUDE.md with the v2.3 `blueprint-sync.md` in place. The v2.3 `blueprint-sync.md` adds two cascade rows ("Query Routing Rule change" and "Ops file change copied by step g") that are irrelevant in a v2.2 context but harmless. No behavioral system failure; no data loss; the extra cascade rows are informational guidance for developers only.

**Recommendation**
Add a backup step and a rollback entry for `blueprint-sync.md`:

In the "On approval, execute in order" section, add after step b2:
```
b3. Copy `scheduled-tasks/ops/blueprint-sync.md` → `backups/blueprint-sync.md-v2.2-<YYYY-MM-DD>.bak`
```

In the Rollback section, extend the v2.2→v2.3 bullet:
```
v2.2→v2.3: Rename backups/CLAUDE.md-v2.2-<date>.bak → CLAUDE.md; rename backups/conventions.md-v2.2-<date>.bak → scheduled-tasks/ops/conventions.md; rename backups/blueprint-sync.md-v2.2-<date>.bak → scheduled-tasks/ops/blueprint-sync.md; revert Schema: in hot.md to v2.2.
```

| Field | Value |

| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `template/scheduled-tasks/ops/reference.md` |
| **Location** | Directory Structure section — blueprint/ subtree |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> The `blueprint/` subtree in `reference.md` lists: `LICENSE`, `.gitignore`, `README.md`, `setup-guide.md`, `user-guide.md`, `troubleshooting.md`, `CHANGELOG.md`, `ROADMAP.md`, `template/`, and `skills/`.

The `docs/` directory (containing `audit-report-template.md` and the superpowers design/plan files) is absent from the subtree. `.gitignore` does not exclude `docs/` — it is distributed to end users as part of the blueprint clone.

**Criteria**
The `reference.md` Directory Structure section is the canonical diagram consulted by the agent when a user asks about directory layout or file locations. It should accurately represent all directories that are part of the distribution. `docs/` is committed to the blueprint repository, not gitignored, and therefore present in any clone of the blueprint. Its absence from the blueprint/ subtree means an agent or developer consulting reference.md for structure information would be unaware of `docs/` and its contents. Note: `docs/` is shown at the working-folder root level in the same diagram (labeled "Developer use only"), but this position corresponds to the blueprint-authoring scenario where the repository root IS the working folder — it does not represent the deployed-user scenario where `docs/` lives inside `blueprint/`.

**Cause**
The `docs/` directory was likely added after `reference.md`'s blueprint/ subtree was last reviewed, or was considered a developer-only artifact not worth documenting for users. AUD-031 added `ROADMAP.md` to the blueprint/ subtree; `docs/` was not included at that time.

**Consequence**
Documentation-only impact. An agent reading reference.md to answer a user's directory-structure question would report an incomplete blueprint/ tree, omitting `docs/`. A developer unfamiliar with the repository who consults reference.md before navigating the blueprint would not find `docs/` in the blueprint/ subtree and might be confused if they see the actual directory.

**Recommendation**
Add `docs/` to the blueprint/ subtree in `reference.md`, after `ROADMAP.md` and before `template/`:

```
│   ├── ROADMAP.md
│   ├── docs/                   ← Audit report template and design specs. Developer use only.
│   │   └── audit-report-template.md
│   ├── template/
```

(Sub-files under `docs/superpowers/` are numerous and evolve quickly — listing only `audit-report-template.md` keeps the diagram stable while covering the schema-critical file.)

### [AUD-2026-04-25-038] `docs/audit-report-template.md` risk-level derivation is self-contradictory

| Field | Value |
|---|---|
| **ID** | AUD-2026-04-25-038 |
| **Severity** | STYLE |
| **Status** | RESOLVED |
| **File** | `docs/audit-report-template.md` |
| **Location** | Executive Summary section — Overall Risk derivation note |
| **Owner** | — |
| **Target Date** | — |

**Condition**
> `Risk is derived from the highest-severity open finding: any unresolved CRITICAL → CRITICAL; all resolved CRITICALs but open WARNINGs → HIGH; all resolved WARNINGs but open STYLEs → MEDIUM; only resolved or STYLE findings → LOW; zero findings → CLEAN.`

The derivation contains two contradictory clauses for the case where the only open findings are STYLE severity:
- Clause 3: "all resolved WARNINGs but open STYLEs → MEDIUM" — open STYLEs → MEDIUM
- Clause 4: "only resolved or STYLE findings → LOW" — STYLE findings → LOW

Both clauses apply when all findings are STYLE-severity. They produce different risk levels.

**Criteria**
A rule set that maps the same input to two different outputs is logically inconsistent. An agent or developer following this derivation for a STYLE-only audit has no unambiguous answer. Observed convention (AUD-008, AUD-009) uses LOW for STYLE-only reports, suggesting clause 4 is the intended rule and clause 3 is either a leftover or applies to a scenario that clause 4 was meant to supersede.

**Cause**
The risk-level derivation appears to have been written incrementally. Clause 3 ("open STYLEs → MEDIUM") may have been the original rule, and clause 4 ("only resolved or STYLE findings → LOW") was added later to reflect the intent that STYLE findings are documentation-only and should not elevate risk. The earlier clause was not removed when the new one was added.

**Consequence**
Documentation-only impact. An auditor following the template for a STYLE-only report sees two conflicting rules and must pick one. Inconsistent risk levels across reports make trend analysis harder. Actual audits in this system have consistently used LOW for STYLE-only (correct per clause 4), so no historical audit reports are retroactively incorrect — but a future auditor could reasonably choose MEDIUM instead and both choices would be "correct" per the template.

**Recommendation**
Remove or clarify the contradictory clause. The simplest fix is to remove clause 3 ("all resolved WARNINGs but open STYLEs → MEDIUM") and replace the derivation with an unambiguous ordered priority list:

Change:
```
Risk is derived from the highest-severity open finding: any unresolved CRITICAL → CRITICAL; all resolved CRITICALs but open WARNINGs → HIGH; all resolved WARNINGs but open STYLEs → MEDIUM; only resolved or STYLE findings → LOW; zero findings → CLEAN.
```

To:
```
Risk is derived from the highest-severity open finding in priority order: (1) any unresolved CRITICAL → CRITICAL; (2) open WARNINGs (no open CRITICALs) → HIGH; (3) only resolved findings and/or open STYLEs → LOW; (4) zero findings → CLEAN. STYLE findings do not elevate risk above LOW — they are documentation-only.
```
